## 🔄 为什么执行顺序是 1 → 3 → 2？

**v7.128版本调整原因**:
- 先确认任务（第1步）
- 再补充关键信息（第2步，原step3）以便生成更精准的维度
- 最后设置雷达图（第3步，原step2）时可基于第2步的回答智能设置默认值

**函数名保留历史命名**:
- `step1_core_task` → 实际第1步 ✅
- `step2_radar` → 实际第3步 ⚠️
- `step3_gap_filling` → 实际第2步 ⚠️

**重要**:
代码中函数名不变（避免大规模重构），但所有日志、注释、文档统一使用**"第1步/第2步/第3步"**来表示实际执行顺序。

## 📝 命名规范

### 后续开发时应遵循：

1. **日志输出**: 统一使用 `[第1步]` / `[第2步]` / `[第3步]`
2. **注释**: 使用 `# 第X步: 功能描述`
3. **文档**: 使用 `第X步 (功能名)` 或 `第X步（函数名）`
4. **payload**: 设置 `step: X` 与实际顺序一致

### 不需要改动：

- ✅ 函数名保持 `step1_core_task` / `step2_radar` / `step3_gap_filling`（避免大规模重构）
- ✅ 函数内部逻辑保持不变
- ✅ goto 参数保持 `progressive_step1/2/3`（workflow注册名）
