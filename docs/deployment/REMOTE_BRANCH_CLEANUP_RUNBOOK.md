# 远程分支清理 Runbook（v8 版本治理配套）

> 目标：清理长期陈旧的远程分支，降低仓库噪音与发布风险；同时保留可回滚锚点（Tag + signoff）。

---

## 1. 清理原则

1. **优先保留发布锚点**：稳定 Tag（`vX.Y.Z`）与发布签字单（`docs/releases/signoff/`）。
2. **先删低风险**：仅删除“已合并到 `origin/main` 且超过阈值”的分支。
3. **禁止直接删主分支**：`origin/main`、`origin/master`、`origin/HEAD` 默认保护。
4. **backup 分支先评估**：若内容已由稳定 Tag 覆盖，可迁移保留策略后删除分支。

---

## 2. 操作流程（标准）

### Step A：预览候选（不删除）

使用脚本 dry-run 获取候选分支：

- `scripts/git_cleanup_remote_branches.py`

默认行为：
- 自动尝试 `fetch --prune`
- 输出远程分支盘点（分支名/天龄/是否合并）
- 输出可删除候选（仅已合并 + 超龄）
- **不执行删除**

### Step B：人工审核

审核维度：
- 是否被稳定 Tag 覆盖
- 是否仍被 CI、运维脚本或外部文档引用
- 是否属于应急保留分支（如某次事故复盘对应分支）

### Step C：执行删除

审核通过后，使用 `--apply` 执行删除。

建议阈值：
- 日常清理：`14` 天
- 月度大扫除：`30` 天

### Step D：清理后验证

- 再次执行 dry-run，确认候选清零或显著减少
- 在发布周报/运维记录中留痕

---

## 3. 风险分级

### 低风险（可直接删）

- 已合并到 `origin/main`
- 超过阈值
- 不在保护列表

### 中风险（先评估再删）

- 未合并但长期冻结
- `backup-*`、`clean-*` 等历史临时分支

### 高风险（默认不删）

- `origin/main` / `origin/master`
- 正在被发布、热修、应急流程使用的分支

---

## 4. 与版本治理 SOP 的关系

建议与以下机制联动：

- 版本策略：`docs/releases/VERSIONING_POLICY.md`
- 快速启动SOP：`QUICKSTART.md`
- 紧急回滚：`EMERGENCY_RECOVERY.md`
- 回滚脚本：`scripts/rollback_to_tag.py`

核心思想：
- **分支可以清理，发布锚点不能丢**。

---

## 5. 建议频率

- 每周：轻量清理（阈值14天）
- 每月：全量审计（阈值30天，含 backup 分支复核）
