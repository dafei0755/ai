# 代码质量标准

> 质量标准是判断"是否可以合并"的客观依据，而非追求完美的理由。
> 以下标准与使用何种工具无关，工具会随时代演进，标准保持稳定。

---

## 后端（Python）

### 代码风格

- **行长度上限**：100 字符
- **缩进**：4 个空格，禁止 tab
- **导入顺序**：标准库 → 第三方库 → 项目内部模块，各组之间空行分隔
- **命名**：函数和变量使用 `snake_case`，类使用 `PascalCase`，常量使用 `UPPER_SNAKE_CASE`

### 类型系统

- 所有公共函数（模块对外暴露的接口）必须有完整的参数类型和返回值类型注解
- 内部私有函数（以 `_` 开头）建议类型注解，不做强制要求
- 不得使用裸 `Any` 作为返回类型，若必须使用需添加注释说明原因
- Pydantic 模型字段必须有类型，且对外接口字段建议有 `description`

### 复杂度

- 单个函数不超过 50 行（不含注释和空行）
- 单个类不超过 300 行
- 嵌套层级不超过 4 层（超过应提取为独立函数）
- 圈复杂度建议不超过 10（超过应拆分逻辑）

### 安全

- 禁止将密钥、Token、密码硬编码在代码中
- 禁止将 `.env` 文件提交到版本库
- 用户输入必须经过验证，不得直接拼接进 SQL 或系统命令
- 安全扫描无高危（High）级别漏洞

## 前端（TypeScript / React）

### 代码风格

- **行长度上限**：100 字符
- **缩进**：2 个空格
- **引号**：字符串使用单引号，JSX 属性使用双引号
- **分号**：不加结尾分号（与 Prettier 默认配置保持一致）

### 类型系统

- 禁止使用 `any`，确实需要宽松类型时使用 `unknown` 并做类型守卫
- Props 必须定义 TypeScript 接口，不得使用隐式 `any`
- API 返回数据必须有类型定义，不得裸用 `response.data`

### 组件规范

- 使用函数式组件 + Hooks，禁止新增 Class 组件
- 组件文件名与导出类名一致，使用 `PascalCase`
- 副作用（数据获取、订阅）放在 `useEffect` 中，不在渲染路径中产生副作用
- 每个组件单一职责：UI 渲染组件不应包含业务逻辑，业务逻辑抽象为 custom hook

### 性能

- 避免在渲染函数中创建新对象/数组作为 props（会触发不必要重渲染）
- 大数据列表使用虚拟化或分页，不做全量渲染
- 图片资源使用 next/image 或指定尺寸，避免布局偏移

## 通用标准

### 注释原则

注释解释"为什么"，不解释"是什么"。代码本身应该足够清晰，不需要逐行解释它在做什么。

```
✅ # 绕过 OpenAI 速率限制，每批次间隔 1 秒
❌ # 等待 1 秒
```

特殊处理（临时方案、已知限制、待解决问题）必须注释，注明原因和计划：

```python
# TODO(v9.0): 待 LangGraph 0.3 稳定后移除此兼容层
# 参见: https://github.com/langchain-ai/langgraph/issues/xxx
```

### 错误处理

- 不得静默吞掉异常（空的 `except` 或 `catch` 块）
- 捕获异常后，要么处理（恢复/降级），要么记录日志后重新抛出
- 用户可见的错误信息不得暴露内部堆栈或系统路径

### 废弃代码

- 确认不再使用的代码立即删除，不留注释掉的代码块
- 废弃的 API 路由使用明确的废弃标记并设定移除版本
- 临时的调试代码（print、console.log）在提交前全部清除

## 质量门禁（合并前必须通过）

| 检查项 | 通过标准 |
|-------|---------|
| 静态分析 | 无新增 Error 级别违规 |
| 类型检查 | 无新增类型错误 |
| 安全扫描 | 无高危漏洞 |
| 单元测试 | 全部通过，0 FAILED |
| 测试覆盖率 | 不低于当前基线（不允许覆盖率下降） |
| 前端 lint | 0 Error（Warning 允许，但需有计划清理） |
| 根目录清洁度 | 无临时脚本文件（`_*.py`、`*.txt` 结果文件等） |
