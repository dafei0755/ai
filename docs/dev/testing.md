# 测试规范

> 测试的目的是**建立信心**：让你和团队相信系统在已知场景下行为正确，并在意外变更时及时预警。
> 测试不是为了凑覆盖率数字，也不是为了完成任务清单。

---

## 测试分层与职责

### 单元测试（Unit）

**验证什么**：单个函数或类在隔离状态下的行为。

**特点**：
- 不依赖外部服务（数据库、Redis、LLM API）
- 执行速度快（单个测试 < 100ms）
- 用 Mock 替代所有 I/O 边界

**适合的场景**：
- 数据转换、计算逻辑
- 条件分支覆盖（正常路径 + 边界值 + 异常路径）
- Pydantic 模型验证逻辑
- 工具函数

**不适合的场景**：
- 验证多个模块协作的效果
- 验证与真实外部服务的交互

---

### 集成测试（Integration）

**验证什么**：多个模块组合后的行为，以及与外部依赖的交互契约。

**特点**：
- 可以连接真实的本地服务（如 Redis），但不调用付费 API
- 执行速度中等（单个测试可接受 1-10 秒）
- 关注接口契约，而非内部实现

**适合的场景**：
- API 路由 → 服务层 → 数据层的完整路径
- LangGraph 工作流节点间的状态流转
- 缓存层的读写一致性
- 错误降级路径（主路径失败时的兜底行为）

---

### 端到端测试（E2E）

**验证什么**：从用户视角出发的完整业务流程。

**特点**：
- 尽量使用真实环境（或接近真实的 staging 环境）
- 执行慢，数量少（只覆盖核心路径，不追求全覆盖）
- 发现的问题往往是集成测试覆盖不到的边缘情况

**适合的场景**：
- 完整的问卷填写 → 报告生成流程
- 用户登录 → 项目分析 → 结果展示
- 关键 API 从请求到响应的完整链路

---

### 回归测试（Regression）

**验证什么**：历史上出现过的 bug 不再复现。

**特点**：
- 每个历史 bug 对应一个或多个回归测试用例
- 测试用例描述中应包含 bug 来源（版本、issue）
- 永久保留，绝不因"代码已重构"而删除

---

## 测试文件生命周期

### 四象限判断

|  | 接口存在 | 接口不存在/已废弃 |
|--|---------|----------------|
| **测试通过** | ✅ 正常维护 | ⚠️ 立即排查原因 |
| **测试失败** | 🔴 必须修复才能合并 | 📦 归档到 `tests/_archive/` |

### 规则一：测试与实现同步

功能代码和对应测试在同一次提交落地。

唯一例外是 TDD 先写测试的情况，此时测试文件顶部必须声明：

```python
"""
⚠️  TDD ASPIRATIONAL TEST
功能未实现，计划版本：vX.Y
若超过 2 个迭代未实现，执行归档。
"""
pytestmark = pytest.mark.skip(reason="aspirational: 待 vX.Y 实现")
```

### 规则二：架构变更后立即审查相关测试

以下操作完成后，必须在同一 PR 中处理受影响的测试文件：

| 变更类型 | 需要审查的测试范围 |
|---------|----------------|
| 删除模块/函数/类 | 所有 import 该符号的测试文件 |
| 重命名接口 | 所有调用旧接口名的测试文件 |
| 修改 State 字段 | 所有依赖该字段的集成/端到端测试 |
| 废弃 Feature Flag | 依赖该 Flag 分支的测试 |

### 规则三：废弃测试立即归档

不再对应任何现存实现的测试，立即移至 `tests/_archive/`，并在文件头注明原因。不允许让失败的废弃测试留在主测试目录污染报告。

## 覆盖率原则

- 覆盖率是结果，不是目标。不为提高数字而写无意义的测试。
- 每次 PR 不允许覆盖率低于合并前的基线（只能提升或持平）。
- 核心业务逻辑（agents、workflow、services）覆盖率目标 ≥ 80%。
- 工具函数、配置文件、迁移脚本不强制要求覆盖率。

---

## 测试标记（Markers）

给测试打上恰当的 marker，使 CI 和本地开发都能精准选择运行范围：

| Marker | 含义 |
|--------|------|
| `unit` | 单元测试，无外部依赖 |
| `integration` | 集成测试，可能需要本地服务 |
| `e2e` | 端到端测试 |
| `regression` | 回归测试 |
| `llm` | 需要真实 LLM API 调用（默认跳过） |
| `slow` | 执行时间 > 5 秒（CI 中可选跳过） |
| `redis` | 需要 Redis 连接 |

快速验证时（本地开发、PR 草稿阶段）跳过慢速测试：

```
pytest -m "not llm and not slow"
```

---

## 测试命名规范

测试函数名应完整描述**场景**和**预期结果**，而非描述实现：

```
✅ test_questionnaire_skips_followup_when_score_above_threshold
✅ test_redis_session_returns_none_on_miss
❌ test_session_1
❌ test_get
```

测试失败时，函数名本身应足以告诉你"什么场景出了什么问题"，无需阅读测试体。
