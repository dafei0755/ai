# 标准开发流程

> 每一次功能开发、缺陷修复或架构调整，都应经历以下五个阶段。
> 每个阶段有明确的完成状态，到达该状态才进入下一阶段。

---

## 阶段一：需求理解

**目标**：清晰定义"要解决什么问题"，而不是"要实现什么功能"。

### 应达到的状态

- 能用一句话描述问题的根因（而非表象）
- 明确影响范围：哪些模块、哪些用户路径会受到影响
- 明确成功标准：如何判断这个问题被解决了

### 常见陷阱

- 直接从症状跳入实现，忽略根因分析
- 将多个独立问题混入同一次变更
- 成功标准模糊（"感觉好多了"不算标准）

---

## 阶段二：方案设计

**目标**：在动手之前，明确做什么、不做什么、风险在哪里。

### 应达到的状态

- 确定变更范围：涉及哪些文件/模块/接口
- 评估风险：是否影响已有功能、是否有回滚路径
- 确定验证方式：完成后如何证明它是对的
- 方案已被确认（自己或同伴的理性审查，不是敷衍地点头）

### 常见陷阱

- 跳过方案设计，依赖"边做边想"
- 对影响范围估计过于乐观（认为"只改一行"）
- 没有想好回滚方案就开始执行高风险变更

---

## 阶段三：实现

**目标**：按方案完成变更，保持专注，不扩散。

### 应达到的状态

- 只做方案设计中已确认的事
- 每个子任务独立完成，可单独验证
- 分支干净：提交信息清晰，无调试代码残留

### 实施原则

**分支管理**

任何代码变更必须在独立分支上完成。分支命名体现目的：

| 类型 | 前缀示例 |
|------|---------|
| 缺陷修复 | `fix/` |
| 新功能 | `feature/` |
| 重构 | `refactor/` |
| 架构调整 | `arch/` |
| 清理/工具 | `chore/` |

单个分支生命周期建议不超过 1 周。超过则说明任务过大，需要拆分。

**提交信息**

遵循 [Conventional Commits](https://www.conventionalcommits.org/zh-hans/)：

```
<type>(<scope>): <描述>
```

type 白名单：`fix` / `feat` / `refactor` / `arch` / `chore` / `docs` / `test`

好的提交信息描述的是**意图**，不是操作：

```
✅ fix(settings): 为 DEBUG 字段添加 bool 容错，防止导入期崩溃
❌ fix: 修了个 bug
❌ update
```

### 常见陷阱

- 实现过程中"顺手"修了不相关的问题，导致分支职责扩散
- 提交信息无信息量，一个月后无法溯源
- 调试代码（print、临时注释）忘记清理就提交

---

## 阶段四：验证

**目标**：用客观证据证明变更达到了阶段一定义的成功标准。

### 应达到的状态

- 单元测试全部通过，无 FAILED / ERROR
- 相关集成测试通过
- 代码质量检查无新增违规
- 人工验证关键路径行为符合预期

### 验证层次

| 层次 | 验证内容 | 必要性 |
|------|---------|--------|
| 单元测试 | 被修改的函数/类行为正确 | 必须 |
| 集成测试 | 模块间交互正常 | 涉及接口变更时必须 |
| 端到端测试 | 完整用户路径不受影响 | 核心路径变更时必须 |
| 代码质量 | lint / type check 无新增问题 | 必须 |
| 人工验证 | 启动服务，手动走一遍关键路径 | 可见行为变更时必须 |

### 常见陷阱

- 只跑了自己写的新测试，没有跑回归测试套件
- "本地没问题"替代了系统性验证
- 测试通过但覆盖的是错误场景

---

## 阶段五：合并

**目标**：将干净、经过验证的变更安全并入主干。

### 应达到的状态

- 测试套件全绿（0 failed, 0 errors），或所有跳过均有合理声明
- 提交历史清晰，无多余的 merge commit 或 WIP 提交
- 相关文档已同步更新（若有接口/行为变更）
- CHANGELOG 已记录（若为可发布变更）

### 常见陷阱

- 带着失败测试合并，准备"下次再修"
- 文档未更新，导致文档与实现不一致
- 遗漏了 CHANGELOG 更新，发布时无法追踪变更历史

---

## 紧急修复流程

时间紧急时，流程可以压缩，但不可以跳过验证：

```
快速定位根因 → 最小化修复 → 验证核心路径 → 合并 → 补充测试覆盖
```

"最小化修复"的含义：只改动必要部分，不在紧急修复中顺带重构。
