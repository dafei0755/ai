# Step 2 搜索任务数量验证与自动补充机制 - v7.420

## 🎯 问题描述

**问题来源**：用户反馈截图显示step2搜索任务清单存在以下问题：
1. step1的搜索框架有5条，每条都有清晰的搜索方向
2. step2搜索任务清单的每个分页对应一个搜索框架
3. 但有些搜索清单过少，有些为空
4. 不满足"不可少于step1搜索框架的方向，并智能扩展，更加详细"的要求

## 🔍 根本原因分析

1. **Prompt强度不足**：
   - 虽然prompt中包含数量要求，但LLM可能不够重视
   - 缺少明确的警告和验证清单
   - 没有强调"绝对禁止"的严重性

2. **缺少验证机制**：
   - 生成后没有验证每个板块的任务数量
   - 没有自动补充机制
   - 完全依赖LLM的一次性生成

## ✅ 修复方案

### 1. 强化 Prompt 数量约束（v7.420）

**文件**：`intelligent_project_analyzer/config/prompts/step2_search_task_decomposition.yaml`

**修改内容**：
```yaml
### 3. 数量约束（❗❗❗极其重要，必须严格遵守❗❗❗）

**硬性要求（不可违反）**：
- **每个板块至少 3 个搜索任务**（绝对不能少于3个）
- **推荐每个板块 5-8 个搜索任务**（充分挖掘多维度）
- **总数下限**：搜索任务总数 ≥ 板块数 × 3
- **绝对禁止**：
  1. 任何板块的搜索任务为空
  2. 任何板块的搜索任务少于3个
  3. 只照搬框架子项，未进行深度挖掘
  4. 所有板块的搜索任务数量相同（应该根据复杂度差异化）

**数量计算示例**：
- 5个板块 → 最低15个（5×3），推荐25-40个（5×5-8）
- 4个板块 → 最低12个（4×3），推荐20-32个（4×5-8）
- 3个板块 → 最低9个（3×3），推荐15-24个（3×5-8）

**验证清单**（生成后必须自查）：
✅ 每个板块都有搜索任务？
✅ 每个板块至少3个任务？
✅ 搜索任务比框架子项更详细、更具体？
✅ 涵盖了理论、案例、技术、用户、趋势等多维度？
✅ 复杂板块的任务数量明显多于简单板块？
```

**关键改进**：
- 使用❗符号和加粗强调重要性
- 明确列出"绝对禁止"的行为
- 提供验证清单，引导LLM自查
- 增加推荐数量从4-6提升到5-8

### 2. 添加验证和自动补充机制（v7.420）

**文件**：`intelligent_project_analyzer/services/four_step_flow_orchestrator.py`

**新增方法**：

#### 2.1 `_validate_and_supplement_queries`
```python
async def _validate_and_supplement_queries(
    self,
    queries: List[SearchQuery],
    output_framework: OutputFramework,
    user_input: str,
) -> List[SearchQuery]:
    """
    验证搜索任务数量并自动补充不足的任务

    核心逻辑：
    1. 统计每个板块的搜索任务数量
    2. 如果某个板块任务数少于3个，自动生成补充任务
    3. 确保所有板块都有充分的搜索覆盖
    """
```

**功能**：
- 统计每个板块的搜索任务数量
- 识别任务数量不足的板块（<3个）
- 为不足的板块生成补充任务
- 合并原始任务和补充任务

#### 2.2 `_generate_supplementary_queries_for_block`
```python
async def _generate_supplementary_queries_for_block(

    """
    为单个板块生成补充搜索任务

    - 避免与现有任务重复
```

## 🔧 技术特点

### 1. 渐进式增强
- 首先依赖强化的Prompt引导LLM生成充足任务
- 其次通过验证机制兜底，自动补充不足
- 最后通过降级策略确保系统鲁棒性

### 2. 智能补充
- 分析现有任务，避免重复
- 基于6大维度（理论、案例、技术、用户、趋势、对比）
- 包含用户特征，保持任务的针对性

### 3. 系统鲁棒性
- LLM补充失败时自动降级到预定义模板
- 确保即使在极端情况下也能满足最低要求
- 日志清晰，便于调试和优化
