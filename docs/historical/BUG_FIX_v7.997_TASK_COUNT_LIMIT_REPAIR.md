# v7.997 任务数量限制修复报告

## 问题描述

**用户反馈**: 目前所有测试的任务梳理数量（问卷第一步）都没有超过13个，13应该是底线而不是上限。

**案例17（中国书法大酒店）**:
- 项目规模: 50000㎡，250间客房
- 功能复杂度: 住宿+会议+展览+学术交流，多维度文化表达
- 预期任务数: 23-36个（复杂项目）
- 实际生成: 仅13个 ❌

## 根因分析

### 问题定位

通过排查发现，虽然 `core_task_decomposer.py` 代码已升级到 v7.997 标准（MIN_TASKS=8, MAX_TASKS=52），但 **Prompt配置文件未同步更新**。

### 关键证据

**文件**: `intelligent_project_analyzer/config/prompts/core_task_decomposer.yaml`

**问题点1 - business_config（第14-22行）**:
```yaml
# ❌ 旧版配置（v7.970）
business_config:
  min_tasks: 3   # 应为 8
  max_tasks: 12  # 应为 52
  base_tasks: 5  # 应为 13
```

**问题点2 - system_prompt任务数量说明（第182-191行）**:
```yaml
# ❌ 旧版描述
- 系统会根据输入复杂度智能评估建议的任务数量范围（3-12个）
- **简单需求**（复杂度<0.3）：生成 **3-5 个任务**
- **中等需求**（复杂度0.3-0.6）：生成 **5-8 个任务**
- **复杂需求**（复杂度>0.6）：生成 **8-12 个任务**
```

### 影响链路

```
Prompt配置文件(3-12)
    ↓
LLM读取system_prompt
    ↓
生成任务时被约束在旧范围
    ↓
即使代码传入recommended_max=36，LLM仍按Prompt生成8-12个
    ↓
最终任务数限制在13个以下 ❌
```

## 修复方案

### 修改内容

**1. 更新 business_config**

```yaml
# ✅ v7.997 新配置
business_config:
  min_tasks: 8   # 简单需求底线（+33%）
  max_tasks: 52  # 超大型项目上限（+30%）
  base_tasks: 13 # 中等复杂度基准（+30%）
```

**2. 更新 system_prompt 任务数量说明**

```yaml
# ✅ v7.997 新标准
- **简单项目**（复杂度 0-0.15）：生成 **8-13 个任务**
  - 示例：单一功能住宅设计、明确单一主题的小型项目
  - 要求：基础调研 + 核心任务覆盖

- **中等项目**（复杂度 0.15-0.30）：生成 **13-23 个任务**
  - 示例：多维度考虑的常规项目、单体建筑设计
  - 要求：全面调研 + 多维度分析

- **复杂项目**（复杂度 0.30-0.50）：生成 **23-36 个任务**
  - 示例：多对立概念、跨界融合、深厚文化背景、大型综合体
  - 要求：深度调研 + 多对象细粒度拆解

- **超大型项目**（复杂度 0.50-1.0）：生成 **36-52 个任务**
  - 示例：城市更新、文化地标、超大面积复合功能项目
  - 要求：极致深度 + 全面覆盖 + 独立关键对象任务
```

## 已知限制与后续优化

### 1. 超大型项目识别

**当前问题**: 狮岭村民宿集群（多建筑师对标、文化深度）仅得到0.362复杂度，未达到超大型阈值0.50。

**原因**:
- 缺少量化指标（面积、客房数、投资额）
- structured_data 未充分传递项目规模信息
- 复杂度算法主要依赖文本分析，对隐式复杂度捕捉不足

**优化方向**:
- 增强 requirements_analyst 对规模指标的提取
- 调整复杂度算法权重（文化深度+跨界融合应加分）
- 考虑将超大型阈值从0.50降至0.45

### 2. LLM遵循率监控

**监控指标**:
- LLM实际生成任务数 vs 推荐范围
- 任务粒度质量（是否符合v7.970标准）
- 任务类型分布（research/analysis/design比例）

**建议**:
- 收集前50个session的任务生成数据
- 分析LLM对新Prompt的响应准确率
- 必要时进一步调整Prompt措辞
