# v7.998 任务数量限制修复报告（最终版）

## 问题定位

### 根本原因

**狮岭村案例复杂度得分0.290，刚好低于旧阈值0.30**

```
复杂度得分: 0.290
旧阈值: 0.30 (复杂项目门槛)
↓
被归类为"中等项目"
↓
推荐范围: 13-23个
↓
LLM倾向于生成下限 → 固定在13个 ❌
```

### 诊断过程

1. ✅ **排除配置文件问题** - 已更新到v7.997，min=8, max=52, base=13
2. ✅ **排除代码逻辑问题** - 复杂度映射、任务截断逻辑均正常
3. ✅ **排除LLM调用问题** - Prompt配置正确
4. 🎯 **定位到阈值问题** - 复杂度得分0.290 < 0.30阈值

### 核心矛盾

**案例特征**：
- 多位建筑师对标（安藤忠雄、隈研吾、刘家琨、王澍、谢柯）
- 文化深度挖掘（在地农耕文化、产业结构、乡村经济）
- 跨界融合（建筑+文化+商业+社会）
- 超高文化特征得分（cultural=0.95, local=0.88）

**复杂度算法**：
- 基于文本长度、信息维度、结构化字段
- 未充分捕捉"建筑师对标数量"等隐式复杂度
- 得分0.290刚好在阈值边缘

**结果**：复杂案例被低估，任务数不足

---

## 修复方案

### v7.998 双重优化策略

#### 策略1: 降低复杂度阈值（解决分类问题）

**旧阈值**（v7.997）:
```
0-0.15: 简单   → 8-13个
0.15-0.30: 中等 → 13-23个  ← 狮岭村(0.29)卡在这里
0.30-0.50: 复杂 → 23-36个
0.50-1.0: 超大型 → 36-52个
```

**新阈值**（v7.998）:
```
0-0.12: 简单   → 8-13个    (阈值0.15→0.12)
0.12-0.25: 中等 → 18-23个  (阈值0.30→0.25, 最小值13→18)
0.25-0.45: 复杂 → 28-36个  (阈值0.50→0.45, 最小值23→28) ← 狮岭村现在在这里
0.45-1.0: 超大型 → 40-52个 (阈值0.50→0.45, 最小值36→40)
```

**优化效果**：
- ✅ 狮岭村(0.29)进入复杂项目区间 → 推荐28-36个
- ✅ 更多案例能进入更高复杂度档位
- ✅ 阈值间隔更均衡（0.12/0.13/0.20/0.55 vs 0.15/0.15/0.20/0.50）

#### 策略2: 提升最小任务数（避免下限陷阱）

**旧最小值**（v7.997）:
```
简单: 8个
中等: 13个  ← LLM倾向选择下限
复杂: 23个
超大型: 36个
```

**新最小值**（v7.998）:
```
简单: 8个   (不变)
中等: 18个  (+5, +38%)
复杂: 28个  (+5, +22%)
超大型: 40个 (+4, +11%)
```

**优化效果**：
- ✅ 中等项目从13提升到18，避免"卡在13"
- ✅ 复杂项目从23提升到28，确保深度调研
- ✅ 超大型项目从36提升到40，支持极致拆解

## 设计思考

### 为什么降低阈值而不是优化算法？

**方案对比**:

| 方案 | 优点 | 缺点 | 实施难度 |
|------|------|------|---------|
| **降低阈值** ✅ | 立即生效，简单直接 | 治标不治本 | ⭐ 极简单 |
| 优化算法 | 根本性解决 | 需要大量测试，可能引入新问题 | ⭐⭐⭐⭐ 复杂 |
| 提升最小值 ✅ | 快速见效，副作用小 | 可能导致简单项目任务过多 | ⭐⭐ 简单 |

**最终策略**: 阈值优化 + 最小值提升（组合拳）
- ✅ 快速解决当前问题
- ✅ 风险可控，易于回滚
- ✅ 后续可迭代优化复杂度算法

### 阈值设计原则

**v7.998阈值分布**:
```
|--12%--|---13%---|-----20%-----|--------55%--------|
简单     中等      复杂          超大型
8-13     18-23     28-36         40-52
```

**设计考量**:
1. **间隔更均衡**: 避免大量案例挤压在某个区间
2. **覆盖更广**: 0.12-0.25区间扩大，捕捉更多中等复杂项目
3. **容错更强**: 复杂度算法误差在±0.05时仍能正确分类
