## 📋 问题背景

### v7.999版本遗留问题

在v7.999修复了Few-shot Learning trap后，用户提出两个关键质疑：

#### 问题1：补充生成的任务质量如何保证？
- **担忧**：系统性规划 vs 临时补充的碎片化差异
- **观察**：v7.999补充机制虽然激活，但存在解析失败问题
- **原因**：
  - 只展示前10个任务作为参考
  - 补充要求过于简单（仅4条）
  - **未提供JSON格式示例**，导致LLM返回简化格式
  - 缺少粒度、格式、特征比例的一致性约束

#### 问题2：Few-shot示范与推荐数量矛盾
- **现状**：狮岭村示范只有13个任务
- **矛盾**：该案例cultural=0.53, complexity≈0.30，按v7.998应推荐**28-36个任务**
- **危害**：Few-shot示范数量与推荐范围不一致，导致LLM学习错误模式

---

## 🎯 修复方案概览

   - **提供完整JSON schema示例**
   - 强调与原任务保持系统性一致

**2. 重新设计Few-shot示范案例**
   - 将狮岭村示范从13个扩展到**32个任务**
   - 与v7.998推荐范围（28-36）严格对齐
   - 覆盖6个调研阶段：文化基础 → 建筑师理念 → 材料工艺 → 运营体系 → 技术标准 → 综合整合

## ⚠️ 补充任务的质量要求（v7.999.1 强制执行）：

### 1. **粒度一致性**
- 观察已有任务的细分粒度（如"每位建筑师独立成任务"）
- 补充任务必须保持**相同粒度标准**
- ❌ 禁止"多位建筑师""多个案例""等"等混合表述

### 2. **格式规范一致性**
- 标题格式：**动词 + 对象 + 维度清单**
- 研究类任务：必须用"搜索/调研/查找/收集"开头
- 分析类任务：使用"对比/分析/评估/验证"开头
- 每个任务标题罗列**3-5个具体调研维度**

### 3. **系统性扩展策略**
- 优先细分已有任务（如将"材料调研"拆分为"木材/石材/竹材"独立任务）
- 补充缺失的维度（如运营、营销、定价、服务等）
- 补充更多案例/建筑师（每个独立成任务）
- 深化分析层次（如可行性、技术、经济、文化等多维评估）

### 4. **特征比例平衡**
- 观察已有任务的类型分布（research/analysis/output）
- 补充任务应维持类似比例（约60-70% research, 25-35% analysis）
- 保持与项目特征向量的对应关系

### 5. **不重复性检查**
- 每个补充任务必须调研**新的对象或新的维度**
- 与已有任务形成互补，而非重复

## ⚠️ 必须使用**完整JSON格式**（包含所有必需字段）：

```json
{{
  "tasks": [
    {{
      "id": "task_N",
      "title": "搜索/调研 对象 + 维度1、维度2、维度3",
      "description": "详细说明调研的具体内容和目的",
      "source_keywords": ["关键词1", "关键词2", "关键词3"],
      "task_type": "research",
      "priority": "high",
      "dependencies": [],
      "execution_order": N,
      "support_search": true,
      "needs_concept_image": false,
      "concept_image_count": 0
    }}
  ]
}}
```

请**严格遵循以上质量标准和JSON格式**，补充生成{shortage}个新任务。
"""
```

**效果**：✅ LLM返回完整JSON，成功解析并补充到28个任务

## 📝 用户常见问题解答

### Q1: 补充生成的任务质量如何保证？

**A**: v7.999.2通过5大质量标准保证补充任务与原任务系统性一致：
1. **粒度一致性**: 观察原任务粒度，保持相同标准
2. **格式规范**: 动词+对象+维度清单，严格命名规范
3. **系统性扩展**: 细分已有任务，补充缺失维度
4. **特征比例**: 维持research/analysis/output比例
5. **不重复性**: 调研新对象或新维度，形成互补

**实际效果**:
- ✅ 补充机制成功激活（13→28个任务）
- ⚠️ 仍有部分补充任务粒度控制不够理想
- 🔜 后续版本将增加二次验证机制

### Q2: Few-shot示范为什么要从13个扩展到32个？

**A**:
1. **数量一致性**: 狮岭村案例complexity=0.30, 按v7.998应推荐28-36个，示范案例应该符合这个范围
2. **避免Few-shot trap**: 原13个示范导致LLM学习了错误的数量模式
3. **提供正确参考**: 32个任务的完整示范让LLM理解"中等复杂度项目应该拆分到什么程度"
4. **系统性完整性**: 6个阶段32个任务展示了从文化基础到综合整合的完整调研路径

### Q3: 为什么不直接在初始生成时让LLM生成28-36个？

**A**:
1. **LLM行为不可控**: 即使Prompt明确要求28-36个，LLM仍可能受Few-shot影响生成13个
2. **补充机制更可靠**: 代码层强制检查+补充，确保最终数量达标
3. **质量vs数量权衡**: 初始生成13个高质量任务 + 补充15个，比强制一次性生成28-36个效果更好
4. **渐进式优化**: v7.999.2的Few-shot优化会逐步改善初始生成质量
