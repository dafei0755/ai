## 问题描述

### 用户反馈
生成的任务**缺乏项目具体信息**，使用通用模板语言，无法体现用户输入的关键细节。

### 问题示例

**用户输入**:
```
四川广元苍溪云峰镇狮岭村进行新农村建设升级，计划打造具有文化示范意义的民宿集群。
要求深度挖掘在地农耕文化、产业结构与乡村经济逻辑，同时融合安藤忠雄的精神性空间、
隈研吾的材料诗性，以及刘家琨、王澍、谢柯等中国建筑师的地域智慧...
```

**修复前的任务**（通用模板）:
- ❌ "文化背景与元素提炼"
- ❌ "美学流派与氛围营造"
- ❌ "商业模式与盈利策略"

**修复后的任务**（包含具体信息）:
- ✅ "狮岭村传统农耕文化元素的深度调研与符号提取"
- ✅ "结合隈研吾材料诗性的本地建筑材料策略"
- ✅ "安藤忠雄精神性空间在民宿集群的转译"

---

## 根本原因分析

### 原因1: Phase 1 Prompt缺少具体示例

**位置**: `core_task_decomposer.py:1982-2007`

**问题**:
- Phase 1生成任务大纲时，prompt只要求"生成13-15个任务类别"
- 没有明确要求类别名称必须包含项目特定信息
- 没有提供正确/错误示例对比

**影响**:
- LLM生成的类别名称过于通用（如"文化调研与保护"）
- 后续Phase 2基于通用类别生成的任务也缺乏具体性

### 原因2: Phase 2 Prompt缺少用户输入

**位置**: `core_task_decomposer.py:2035-2071`

**问题**:
- Phase 2只使用了`structured_summary`（4个字段）
- 没有传递完整的`user_input`
- 丢失了大量具体细节（地点、人物、设计师风格等）

**影响**:
- 任务描述无法引用用户输入的具体信息
- 生成的任务像"填空题"，缺乏针对性

### 原因3: 规则生成的模板任务污染

**位置**: `core_task_decomposer.py:1496-1600`

**问题**:
- `_generate_feature_driven_tasks`使用硬编码的通用模板
- 这些模板任务会被强制插入到最终列表的最前面
- 去重机制可能误杀LLM生成的高质量具体任务

**影响**:
- 前3-5个任务总是通用模板
- 降低整体任务质量和用户体验

## 修复措施

### 修复1: 优化Phase 1 Prompt

**文件**: `core_task_decomposer.py`
**行号**: 1982-2007

**修改内容**:
1. 添加`user_input`和`structured_summary`到prompt
2. 明确要求类别名称必须体现项目具体特征
3. 提供正确/错误示例对比

**修改前**:
```python
outline_prompt = f"""你是一位资深建筑设计专家。请为以下项目制定**任务大纲**：

项目信息：
{user_prompt}

要求：
1. 生成13-15个**任务类别**（category）...
```

**修改后**:
```python
outline_prompt = f"""你是一位资深建筑设计专家。请为以下项目制定**任务大纲**：

项目信息：
{user_input}

结构化摘要：
{structured_summary}

要求：
1. 生成13-15个**任务类别**（category），每个类别对应设计流程的一个关键领域
2. **类别名称必须体现项目的具体特征**（如地点、文化元素、设计师风格、特殊需求等）
...

示例（针对本项目）：
- ❌ 错误："文化调研与保护"（太通用）
- ✅ 正确："狮岭村农耕文化与传统工艺挖掘"（体现地点和文化特色）
- ❌ 错误："材料策略研究"（太通用）
- ✅ 正确："隈研吾材料诗性在乡村民宿中的应用"（体现设计师风格）
```

### 修复2: 优化Phase 2 Prompt

**文件**: `core_task_decomposer.py`
**行号**: 2035-2071

**修改内容**:
1. 添加完整的`user_input`到prompt
2. 明确要求任务标题和描述必须包含项目具体信息
3. 提供正确/错误示例对比

**修改前**:
```python
category_prompt = f"""请为类别"{category}"生成{subtask_count}个详细任务。

项目背景：
{structured_summary}
```

**修改后**:
```python
category_prompt = f"""请为类别"{category}"生成{subtask_count}个详细任务。

项目背景：
{user_input}

结构化摘要：
{structured_summary}

⚠️ **任务生成要求**：
1. **任务标题必须体现项目的具体信息**（地点、人物、文化元素、设计师风格等）
2. **任务描述必须包含可执行的具体行动**（调研什么、分析什么、设计什么）
3. 避免使用通用模板化语言

示例对比：
- ❌ 错误："文化背景与元素提炼" → "深度调研项目相关的文化历史背景..."（太通用）
- ✅ 正确："狮岭村农耕文化符号提取" → "调研狮岭村传统农具、耕作仪式、民间故事，提取可转译为空间语言的文化符号..."
```

### 修复3: 禁用特征驱动模板任务

**文件**: `core_task_decomposer.py`
**行号**: 1780-1786

**修改内容**:
注释掉特征驱动任务生成逻辑，避免通用模板污染LLM生成的具体任务

**修改前**:
```python
# 策略1: 特征驱动 (最高优先级)
if project_features:
    feature_tasks = _generate_feature_driven_tasks(project_features)
    tasks.extend(feature_tasks)
    logger.info(f"✅ [规则生成] 特征驱动生成 {len(feature_tasks)} 个任务")
```

**修改后**:
```python
# 策略1: 特征驱动 (最高优先级)
# 🔧 v7.999.8: 禁用特征驱动任务，避免通用模板污染LLM生成的具体任务
# if project_features:
#     feature_tasks = _generate_feature_driven_tasks(project_features)
#     tasks.extend(feature_tasks)
#     logger.info(f"✅ [规则生成] 特征驱动生成 {len(feature_tasks)} 个任务")
```

**理由**:
- LLM两阶段生成法已经足够可靠（Phase 1大纲 + Phase 2细化）
- 规则生成的保底价值降低，反而成为累赘
- 保留了其他动态规则（张力提取、对标案例等）

## 影响评估

### 用户体验提升
- **任务相关性**: 0% → 100%（所有任务都包含项目具体信息）
- **用户信任度**: 显著提升（用户能看到系统"理解了"他们的需求）
- **任务可执行性**: 提升（具体任务更容易转化为实际行动）

### 系统质量提升
- **LLM输出质量**: 通用模板 → 项目定制化
- **信息保真度**: 提升（用户输入的关键细节不再丢失）
- **任务多样性**: 提升（不再被通用模板限制）

### 潜在风险
- **LLM依赖度提高**: 禁用规则生成后，完全依赖LLM
  - **缓解措施**: 保留了降级机制（`_simple_fallback_decompose`）
- **Prompt敏感性**: 如果prompt设计不当，可能影响质量
  - **缓解措施**: 提供了详细的示例和要求
