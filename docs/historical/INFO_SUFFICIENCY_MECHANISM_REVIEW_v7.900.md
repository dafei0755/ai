## 一、机制本质重新定义

### 1.1 核心目标
**信息充足性检测≠场景完整性评估**
- ❌ 错误理解：要收集所有设计细节才能开始工作
- ✅ 正确理解：判断是否有**最低必要信息**启动需求分析与概念构建

### 1.2 判断标准演进

**v7.800之前（7维度）**：
- 侧重"硬信息"：面积、预算、风格
- 缺失：文化深度、利益冲突、技术标准

**v7.900当前（16维度）**：
```python
核心维度（权重0.70）：
1. project_type (项目类型) - 0.15
2. space_constraint (空间约束) - 0.15
3. user_identity (用户身份) - 0.10
4. budget_constraint (预算约束) - 0.10
5. special_health_needs (特殊健康/医疗需求) - 0.10
6. technical_standards (技术规范/专业标准) - 0.10

扩展维度（权重0.25）：
7. functional_needs (功能需求) - 0.08
8. industry_context (行业/领域约束) - 0.08
9. stakeholder_conflicts (利益相关者冲突) - 0.07
10. extreme_environment (极端环境约束) - 0.07
11. multi_stakeholder (多方协同需求) - 0.06
12. cultural_spiritual_context (文化/精神层面) - 0.05
13. style_preference (风格偏好) - 0.05

可选维度（权重0.05）：
14. operational_model (运营模式) - 0.02
15. time_constraint (时间约束) - 0.02
16. regulatory_requirements (法规约束) - 0.01
```

## 五、核心机制原理复盘

### 5.1 为什么是16个（现在建议19个）维度？

**不是越多越好**：
- ❌ 100个维度：过度细分，权重失效，维护成本爆炸
- ❌ 5个维度：过于粗糙，无法捕捉复杂场景
- ✅ 16-20个维度：**认知负荷最优区间** + **实际场景覆盖平衡**

**维度设计哲学**：
1. **必要性原则**: 该维度缺失会导致无法启动设计流程
2. **可检测性原则**: 通过关键词+正则能有效识别
3. **权重合理性**: 反映实际设计流程中的重要程度
4. **互斥性原则**: 维度之间尽量不重叠

### 5.2 阈值设计逻辑

**当前阈值**: 0.50（从v7.800的0.40提升）
**最低元素数**: 3个维度

**为什么0.50合适？**:
```
极简场景（3维度 × 平均0.10权重）= 0.30分 → ❌ 不足
基础场景（5维度 × 平均0.10权重）= 0.50分 → ✅ 充足
复杂场景（8维度 × 平均0.09权重）= 0.72分 → ✅ 充足
```

**P1关键元素动态调整**：
- 检测到关键元素（特殊健康、极端环境等）
- 阈值降低: 每个-0.03，最多-0.10
- 逻辑: 重要信息存在时，其他细节可容忍缺失

### 5.3 智能推断机制（P0优化）

**隐含信息加分（最多+0.3）**:
```python
高净值人群 → +0.20 (预算充足)
特殊需求人群 → +0.15 (功能明确)
大面积项目 → +0.10 (复杂需求)
商业项目 → +0.10 (多维约束)
历史建筑 → +0.15 (结构+法规+文化)
临时项目 → +0.12 (时间+成本+拆卸)
跨文化项目 → +0.10 (文化+标准)
```

**哲学**: 不要求用户说出所有细节，系统能推断常识

## 六、与场景分类系统的协同

### 6.1 信息充足性 vs 场景分类

**信息充足性检测**（capability_detector）：
- 目标: 判断能否启动设计流程
- 输出: is_sufficient (True/False) + score (0-1) + 缺失维度列表

**场景分类系统**（P2优化）：
- 目标: 识别场景类型以路由到专用处理逻辑
- 输出: primary_type + confidence + all_detected_types

### 6.2 两者协同工作流程

```
用户输入
    ↓
1. full_capability_check()
    ├─ deliverable_checks (交付物检测)
    ├─ info_sufficiency (信息充足性) ← 16维度检测
    └─ scenario_classification (场景分类) ← 7类型检测
    ↓
2. 判断逻辑
    ├─ 信息不足 → 进入questionnaire（问卷补全）
    ├─ 信息充足 → 进入phase1（需求分析）
    └─ 场景类型 → 路由到专用prompt模板（未来）
```

### 6.3 未来协同增强点

**可能性1**: 场景类型影响信息充足阈值
```python
# 不同场景类型对信息要求不同
scenario_thresholds = {
    "spatial_design": 0.50,  # 空间设计需要详细信息
    "creative": 0.35,        # 文案创意可以更抽象
    "methodology": 0.40,     # 方法论类中等要求
    "research": 0.45         # 研究类需要清晰目标
}
```

**可能性2**: 信息维度权重根据场景类型动态调整
```python
# 运营管理类场景更看重operational_model
if scenario_type == "operational":
    INFO_ELEMENTS["operational_model"]["weight"] *= 2.0
```
