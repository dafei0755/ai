# 任务数量控制机制深度分析

## 问题1: LLM选择下限不合理 ❌ → ✅ v7.998已修复

### 问题表现

**狮岭村案例**（复杂度0.290）:
- v7.997推荐范围: 13-23个
- LLM实际生成: **13个**（卡在下限）
- 用户预期: 28-36个（多建筑师对标+文化深挖）

### 根本原因分析

#### 1. LLM的保守生成策略

```
Prompt告知: "生成13-23个任务"
        ↓
LLM思维模式: "达到最小要求即可"
        ↓
生成结果: 13个任务（范围下限）
        ↓
问题: 复杂项目调研深度不足
```

**心理学解释**:
- LLM倾向于**满足最低要求**（satisficing而非optimizing）
- 面对范围时，选择**安全的下限**避免生成过多无效任务
- 类似人类行为：任务范围"3-5页报告" → 多数人写3页

#### 2. 复杂度阈值过高（根本问题）

```python
# v7.997阈值设定
if complexity_score < 0.15:   # 简单项目
    return 8-13
elif complexity_score < 0.30: # 中等项目 ← 狮岭村0.290在这里
    return 13-23               # LLM选择13 ❌
elif complexity_score < 0.50: # 复杂项目
    return 23-36
```

**问题链路**:
```
狮岭村特征:
- 5位建筑师对标（安藤忠雄、隈研吾、刘家琨、王澍、谢柯）
- 文化深度挖掘（在地农耕文化、产业结构）
- 跨界融合（建筑+文化+商业+社会）
- cultural=0.95, local=0.88（极高文化特征）

复杂度算法:
- 基于文本长度、信息维度数量、结构化字段
- 未充分捕捉"建筑师数量""文化深度"等隐式复杂度
        ↓
复杂度得分: 0.290
        ↓
阈值判断: 0.290 < 0.30 → 归类为"中等项目"
        ↓
推荐范围: 13-23个
        ↓
LLM生成: 13个（下限）❌
```

### v7.998 双重修复策略

#### 策略1: 降低复杂度阈值（扩大高档位范围）

```python
# v7.998优化后
if complexity_score < 0.12:   # 简单项目（阈值0.15→0.12）
    return 8-13
elif complexity_score < 0.25: # 中等项目（阈值0.30→0.25）
    return 18-23               # 下限13→18
elif complexity_score < 0.45: # 复杂项目（阈值0.50→0.45）← 狮岭村0.290现在在这里 ✅
    return 28-36               # 下限23→28
else:                         # 超大型（阈值0.50→0.45）
    return 40-52               # 下限36→40
```

**修复效果**:
```
狮岭村0.290:
v7.997: 0.290 < 0.30 → 中等项目(13-23) → LLM生成13 ❌
v7.998: 0.290 > 0.25 → 复杂项目(28-36) → LLM生成30+ ✅
```

#### 策略2: 提升最小任务数（避免下限陷阱）

| 项目类型 | v7.997 | v7.998 | 提升幅度 | 设计思路 |
|---------|--------|--------|---------|---------|
| 简单 | 8-13 | 8-13 | 不变 | 简单项目确实不需要太多任务 |
| 中等 | **13**-23 | **18**-23 | +38% | ✅ 避免卡在13（核心修复） |
| 复杂 | **23**-36 | **28**-36 | +22% | 确保深度调研充足 |
| 超大型 | **36**-52 | **40**-52 | +11% | 支持极致拆解 |

**双重保障逻辑**:
```
狮岭村案例（复杂度0.290）:

保障1（降低阈值）:
  0.290 > 新阈值0.25 → 归类为"复杂项目"(28-36) ✅

保障2（提升下限）:
  即使仍归为"中等项目"（阈值判断误差）
  → 最小18个（而非13个）✅
  → 至少增加5个任务（+38%）

结果: 无论归类是否完美，任务数都不会卡在13 ✅
```

### 为什么不优化复杂度算法？

| 方案 | 优点 | 缺点 | 实施难度 | 风险 |
|-----|------|------|---------|-----|
| **降低阈值** ✅ | 立即生效，简单直接 | 治标不治本 | ⭐ 极简单 | 低 |
| **提升下限** ✅ | 快速见效，副作用小 | 简单项目可能任务略多 | ⭐⭐ 简单 | 低 |
| 优化算法 | 根本性解决 | 需要大量测试和数据，可能引入新问题 | ⭐⭐⭐⭐ 复杂 | 高 |

**当前策略**（v7.998）:
- ✅ 组合拳：阈值优化 + 下限提升
- ✅ 风险可控，易于回滚
- ✅ 后续可迭代优化复杂度算法（v8.0+）

### 3. 截断是否会丢失有价值任务？

#### ✅ 不会丢失，有三重保护

**保护1: 按优先级保留**
```python
# 高优先级任务永远优先保留
high_priority_tasks = [t for t in tasks if t.get("priority") == "high"]
other_tasks = [t for t in tasks if t.get("priority") != "high"]
tasks = (high_priority_tasks + other_tasks)[:recommended_max]
```

**示例**:
```
LLM生成42个任务（推荐最大36）:
  high: 10个 → 全部保留 ✅
  medium: 25个 → 保留前23个 ✅
  low: 7个 → 保留前3个（可能丢弃4个low任务）

分析: low优先级任务通常是补充性任务，不会影响核心调研
```

**保护2: 规则强制任务优先级更高**
```python
# v7.995 P2: 规则驱动任务自动标记为高优先级
def _generate_feature_driven_tasks(project_features):
    """根据高分特征（≥0.7）强制生成关键任务"""
    tasks = []
    for feature_id, score in project_features.items():
        if score >= 0.7:
            tasks.append({
                "priority": "high",  # 🔥 强制高优先级
                "source": "rule_feature_driven",
                ...
            })
```

**示例**（狮岭村案例）:
```
project_features高分特征:
- cultural: 0.95 → 生成"文化深度挖掘"任务（high优先级）✅
- commercial: 0.72 → 生成"商业模型研究"任务（high优先级）✅

即使LLM生成的总任务数超过36个，这些规则任务也会被保留
```

**保护3: Prompt明确要求覆盖维度**
```yaml
- 确保覆盖多个维度（对标、文化、客群、愿景、核心矛盾等）
- 🔥 关键要求：如果结构化数据中存在design_challenge（核心张力），
  你必须生成针对这个矛盾的专门研究任务！
```

**结果**: LLM会优先生成关键维度的任务，补充任务放在后面

#### ❌ 可能丢失的任务（低风险）

**丢失场景1**: low优先级的补充性任务
```
示例：
- "收集类似项目的营销文案参考"（low）
- "整理行业常见材料清单"（low）

分析：这些任务通常是辅助性的，不影响核心调研和设计决策
```

**丢失场景2**: LLM生成的重复任务
```
示例：
- Task 12: "调研安藤忠雄的建筑哲学"
- Task 23: "分析安藤忠雄的设计理念"

去重策略: 合并为一个任务，保留描述更详细的版本
```

**丢失场景3**: 与高分特征重复的LLM任务
```
规则任务: "深度挖掘在地农耕文化"（cultural=0.95触发，high优先级）
LLM任务: "调研当地文化元素"（medium优先级）
         ↓
去重后保留规则任务，LLM任务被丢弃（但信息未丢失）✅
```

### 4. 当前机制的优缺点

#### ✅ 优点

**1. 灵活性强**
- Prompt引导为主（80%），截断为辅（20%安全网）
- LLM可以根据具体情况调整任务数（在推荐范围内）

**2. 保护核心任务**
- 高优先级任务永远保留
- 规则驱动任务（高分特征）强制保留
- 关键维度（对标、文化、核心矛盾）必须覆盖

**3. 避免任务泛滥**
- 超过上限时智能截断，防止任务列表过长
- 用户体验更好（28-36个任务合理，不会超过50个）

**4. 可观测性好**
```python
logger.warning(f"⚠️ LLM生成了{len(tasks)}个任务，超过推荐上限{recommended_max}，智能截断")
```
- 日志记录截断事件，便于监控和调优

#### ❌ 缺点（已优化或可接受）

**1. LLM倾向于选择下限** ← v7.998已修复 ✅
- 旧问题：推荐13-23，LLM生成13
- 新策略：提升下限13→18，降低阈值0.30→0.25

**2. 截断可能丢失补充任务** ← 风险可控 ✅
- 丢失的通常是low优先级任务
- 不影响核心调研和设计决策

**3. 复杂度算法不完美** ← 后续优化 🔄
- 当前未充分捕捉"建筑师数量""文化深度"等隐式复杂度
- v8.0计划引入机器学习模型优化

### 5. 截断策略对比

| 策略 | 实施方式 | 优点 | 缺点 |
|-----|---------|------|------|
| **按优先级** ✅ | 保留high > medium > low | 保护核心任务 | 可能丢失low任务 |
| 按时间顺序 | 保留前N个 | 简单直接 | ❌ 可能丢失后面的高优先级任务 |
| 按相似度去重 | 合并重复任务 | 避免冗余 | 需要复杂的语义分析 |
| 让LLM选择 | 要求LLM自己截断 | 更智能 | ❌ LLM可能犯错，不可控 |

**当前采用**: 按优先级 + 适度相似度去重（v7.995混合策略）

### 问题1解决方案：LLM选择下限不合理

**v7.998双重修复** ✅:
```
策略1: 降低复杂度阈值（0.30→0.25）
```
