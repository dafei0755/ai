# 任务梳理与10 Mode Engine的关系分析

## 一、核心关系

**10 Mode Engine（设计模式层）** 是任务梳理生成的**上游决策依据**，而不是直接生成方法。

```
用户输入
  ↓
需求分析（识别设计模式）
  ↓
12维特征向量 + 模式标签
  ↓
任务梳理（根据模式调整任务结构）
  ↓
输出：28-52个任务
```

## 二、10种设计模式与任务生成的映射

### 模式1：概念驱动型（Concept-Driven Mode）

**特征识别**:
- 关键词：精神、表达、身份、文化主题、建筑师风格
- 12维特征：`cultural`高分、`aesthetic`高分

**任务生成策略**:
```
重点类别：
1. 文化背景与元素提炼（5-6个任务）
2. 设计对标与参考研究（4-5个任务）
3. 概念设计与空间策略（5-6个任务）

弱化类别：
- 商业模式与运营策略（1-2个任务）
- 技术系统与材料策略（2-3个任务）
```

**示例任务**:
- "安藤忠雄精神性空间案例研究：建筑哲学、光影运用、清水混凝土美学"
- "项目核心精神主线提炼：从文化母题到空间隐喻"
- "空间叙事路径设计：进入-过渡-高潮-沉静的节奏控制"

## 三、模式识别机制

### 当前实现（隐式识别）

**代码位置**: `core_task_decomposer.py:235-249`

系统通过`project_features`（12维特征向量）隐式识别模式：

```python
def _get_dynamic_guidance(self, project_features: Dict[str, float]) -> str:
    """根据特征向量生成动态指导"""

    # 按分数排序特征
    sorted_features = sorted(project_features.items(), key=lambda x: x[1], reverse=True)

    # 为高分特征（>0.7）提供专项指导
    high_score_features = [(fid, score) for fid, score in sorted_features if score >= 0.7]
```

            'weight': 1.0
        },
        'M10_future_projection': {
            'keywords': ['未来', '趋势', '演进', '长周期', '可适配'],
            'features': {'technical': 0.6, 'professional': 0.6},
            'weight': 1.0
        }
    }

    def detect_mode(self, user_input: str, project_features: Dict[str, float]) -> List[str]:
        """
        识别项目所属的设计模式

        Returns:
            模式列表（按匹配度排序，可能是混合模式）
        """
        mode_scores = {}

        for mode_id, pattern in self.MODE_PATTERNS.items():
            score = 0.0

            # 关键词匹配（权重40%）
            keyword_matches = sum(1 for kw in pattern['keywords'] if kw in user_input)
            score += (keyword_matches / len(pattern['keywords'])) * 0.4

            # 特征向量匹配（权重60%）
            feature_score = 0.0
            for feature_id, threshold in pattern['features'].items():
                if project_features.get(feature_id, 0) >= threshold:
                    feature_score += project_features[feature_id]
            score += (feature_score / len(pattern['features'])) * 0.6

            mode_scores[mode_id] = score

        # 排序并返回得分>0.5的模式
        sorted_modes = sorted(mode_scores.items(), key=lambda x: x[1], reverse=True)
        return [mode_id for mode_id, score in sorted_modes if score > 0.5]

## 四、模式对任务生成的影响

### 影响维度

1. **类别选择**: 不同模式强化不同的标准类别
2. **类别权重**: 调整每个类别的子任务数量（2-5个）
3. **任务粒度**: 某些模式需要更细粒度的任务拆解
4. **任务类型**: research/analysis/design/technical的比例
5. **优先级分配**: high/medium/low的分布

## 五、总结

### 当前状态

- **10 Mode Engine**: 存在于文档层（`sf/10_Mode_Engine`），未直接集成到代码
- **任务生成**: 通过12维特征向量隐式识别模式，但没有显式的模式标签
- **关系**: 特征向量是模式识别的中间层，但缺乏明确的模式→任务映射

### 优化建议

1. **显式模式识别**: 实现`ModeDetector`类，输出明确的模式标签
2. **模式驱动生成**: 根据模式标签调整类别选择和权重
3. **混合模式支持**: 支持多模式并存（如M5+M1）
4. **模式可视化**: 在前端显示识别到的设计模式
5. **模式学习**: 收集用户反馈，优化模式识别准确率

### 价值

- **更精准的任务拆解**: 根据项目本质特征生成任务
- **更清晰的设计逻辑**: 用户理解为什么生成这些任务
- **更专业的咨询能力**: 体现对设计模式的深度理解
- **更好的用户体验**: 任务与项目类型高度匹配

---

**结论**: 10 Mode Engine是任务梳理的**理论基础**和**上游决策依据**，当前通过12维特征向量隐式关联，建议实现显式的模式识别和模式驱动的任务生成策略。
