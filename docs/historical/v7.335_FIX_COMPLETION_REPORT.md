## 📊 修复概览

### 问题描述
历史对话记录间歇性不可见，用户刷新页面后有时能看到，有时看不到。

### 根本原因
开发模式下用户标识符不一致：
- `current_user.user_id = 9999`（数字）
- `current_user.username = "dev_user"`（字符串）
- `get_user_identifier()` 返回值不确定，导致会话过滤逻辑匹配失败

### 修复方案（三层防御）

#### 🥇 层级1：核心修复
**文件**: `intelligent_project_analyzer/api/server.py:367-405`

```python
def get_user_identifier(current_user: dict) -> str:
    if not current_user:
        return "unknown"

    # 🆕 v7.335: 开发模式强制返回固定标识符
    if DEV_MODE:
        return "dev_user"  # 始终返回 "dev_user"

    # 生产模式正常逻辑...
```

**效果**: 开发模式下标识符**100%一致**

#### 🥈 层级2：防御性修复
**文件**: `intelligent_project_analyzer/api/server.py:7415-7445`

```python
# 同时匹配多种可能的标识符
possible_identifiers = [username, str(user_id), "web_user", "dev_user"]
user_sessions = [
    session for session in all_sessions
    if session.get("user_id") in possible_identifiers
]
```

**效果**: 即使标识符不一致，也能找到会话

#### 🥉 层级3：归档会话修复（v7.303已实施）
**文件**: `intelligent_project_analyzer/api/server.py:7848-7867`

```python
# 开发模式返回所有用户的归档会话
if DEV_MODE:
    sessions = await archive_manager.list_archived_sessions(user_id=None)
```

**效果**: 归档会话不受标识符影响

## 🎓 经验总结

### 问题分析方法
1. **现象观察**: 间歇性问题 → 考虑状态/缓存/标识符不一致
2. **代码审查**: 追踪用户标识符的获取和使用路径
3. **时序分析**: 绘制时序图，理解问题发生条件
4. **日志分析**: 查看实际运行时的标识符值

### 修复设计原则
1. **单一真相源**: 固定开发模式标识符为 "dev_user"
2. **防御性编程**: 多层防御，即使核心修复失效也能兜底
3. **向后兼容**: 兼容旧数据（"web_user"、数字ID）
4. **环境隔离**: 开发模式和生产模式行为明确分离

### 测试策略

2. **集成测试**: 模拟完整的会话过滤流程
