## 2. 当前机制基线（必须遵守）

### 2.1 当前真实执行顺序（位于一级大节点“需求分析”内部）

从主计划口径看，完整主线是：

`用户输入 -> 检查门禁 -> 需求分析 -> 任务落实 -> 生成交付结果 -> 追问与迭代`

而本文件关注的“需求分析师真实机制”，指的是一级大节点 **`需求分析`** 内部当前仍然真实在跑的核心子链路，必须按以下顺序理解：

`precheck -> phase1 -> phase2 -> output -> requirements_analyst_review`

关键约束：

1. 这条链路描述的是 **`需求分析` 节点内部的真实执行顺序**，不是面向用户重新平铺的一条一级主线。
2. `Phase2` 当前 **始终执行**，不得按旧理解跳过。
3. `requirements_analyst_review` 是 `需求分析` 内部真实在用的关键子节点，不是可随意忽略的附属展示层。
4. 当前系统输出本质上是：
   - 分析性底稿
   - 任务框架底稿
   - 审阅与交接底稿
5. 这条内部子链的后续消费方向，应继续流向 `output_intent_detection`、信息补全、可选偏好雷达、`questionnaire_summary`，并最终为后续一级大节点 `任务落实` 提供输入。
6. 当前系统 **尚未** 正式显式产出：
   - `FrameworkBundle`
   - `HumanContextProfile`
   - `DecisionTensions`
   - 面向建筑/室内概念方案编制的中间对象包

### 2.2 当前机制的真实能力边界

这里的“能力边界”，指的是当前 `RequirementsAnalystAgentV2` 作为一级大节点 `需求分析` 的内部源头能力边界，不等于整条系统主线已经全部完成重构。

当前需求分析师已经能够：

- 进行 Phase1 快速定性与交付物识别
- 进行 Phase2 L1-L9 深度分析
- 产出 JTBD、核心张力、动机、立场、假设审计、系统性影响、专家叩问
- 在 `requirements_analyst_review` 中展示 27 方法轨迹与质量门禁

当前需求分析师还不能稳定做到：

- 把空间分类结果正式收束为单一主对象
- 把“空间框架 / 模式框架 / ontology 片段 / 参考建筑师方法”正式组装为 `FrameworkBundle`
- 把人本洞察与判断结果收束为显式桥接对象
- 把分析结果自动编译为结构化的概念框架任务书

### 2.3 本次开发的核心目标

本轮开发目标不是重写全部系统，也不是重写主线蓝图，而是把当前已经存在的深度分析能力，组织为更稳定、可消费、可下游接线的对象链：

`ProblemPixels -> SpaceClassification -> FrameworkBundle -> HumanContextProfile -> DecisionTensions -> TaskFramework -> RequirementsInsight`

其中：

- 第 1-5 层对象先作为桥接草案对象推进
- 第 6-7 层对象优先进入正式主链接线

换句话说，本文件要解决的是：

- 如何把一级大节点 `需求分析` 的内部七层装配线收束清楚；
- 如何让该节点在内部形成 `TaskFramework`、在节点收口处形成 `RequirementsInsight`；
- 如何把这些结果稳定交给下游一级大节点 `任务落实`，而不是继续散落在历史字段里。

### 2.4 最短路径实施规范（2026-03-08 生效）

> 目的：在不破坏默认链路与冻结约束的前提下，提供“需求分析最短可运行路径”。

#### 2.4.1 最短路径定义（两种口径）

1. **机制口径最短闭环（默认链路）**
   - `precheck -> phase1 -> phase2 -> output -> requirements_analyst_review`
   - 说明：这是一级大节点 `需求分析` 内部的最小闭环，不含后续二级步骤跳过。

2. **运行时最短路径（Fast Path）**
   - `unified_input_validator_initial`
   - `requirements_analyst`
   - `requirements_analyst_review`
   - `output_intent_detection`
   - `feasibility_analyst`
   - `unified_input_validator_secondary`
   - `project_director`
   - 说明：显式跳过 `progressive_step3_gap_filling`、`progressive_step2_radar`、`questionnaire_summary`。

#### 2.4.2 实施开关与路由原则

1. 开关名称：`ENABLE_REQUIREMENTS_FAST_PATH`
2. 定义文件：`intelligent_project_analyzer/config/feature_flags.py`
3. 默认值：`false`（保持历史行为不变）
4. 生效点：`_unified_input_validator_secondary_node()`
5. 路由原则：
   - 开关关闭：保持主图静态边，继续 `secondary -> progressive_step3_gap_filling`
   - 开关开启：`secondary` 节点返回 `Command(goto="project_director")`，短路覆盖静态边

#### 2.4.3 最短路径步骤 × 组件 × 方法 × 算法对照

#### 2.4.4 Fast Path 跳过项与状态标记

开启 `ENABLE_REQUIREMENTS_FAST_PATH=true` 后，系统必须写入如下标记，确保可观测性与审计可追踪：

- `requirements_fast_path = True`
- `questionnaire_summary_skipped = True`
- `progressive_step3_skipped = True`
- `progressive_step2_skipped = True`

该标记仅表示“流程短路策略生效”，不表示质量门禁天然通过。

#### 2.4.5 与主图冻结约束的一致性

1. 不修改 `main_workflow.py` 的冻结不变量：
   - `unified_input_validator_secondary` 静态下一节点仍保持 `progressive_step3_gap_filling`。
2. 采用“节点内 Command 动态短路”实现最短路径，而非改写主图拓扑。
3. 该策略符合“默认行为不变、按开关启用”的发布安全原则。

#### 2.4.6 适用场景与禁用场景

适用：

1. 高吞吐演示/压测；
2. 输入质量较高、已不需要二级补全追问；
3. 先生成任务分派骨架，再由下游迭代细化。

禁用：

1. 首次陌生项目、缺口明显；
2. 需要完整偏好雷达与需求洞察定稿；
3. 高风险交付前的严谨质量流程。

#### 2.4.7 开发与联调检查清单

1. 开关关闭时，链路必须与历史完全一致；
2. 开关开启时，必须直达 `project_director` 且跳过三个二级步骤；
3. 审阅门禁 `blocker/warn/pass` 行为不得被 fast path 破坏；
4. `requirements_fast_path` 与 `*_skipped` 标记必须可在 state 中观测；
5. 不得引入对 `progressive_step1_core_task` 的任何回退依赖。

#### 2.4.12 最小回归断言（建议）

1. 开关关闭：`workflow` 轨迹必须出现 `progressive_step3_gap_filling`。
2. 开关开启：`workflow` 轨迹必须不出现 `progressive_step3_gap_filling` / `progressive_step2_radar` / `questionnaire_summary`。
3. 开关开启：state 必须包含：
   - `requirements_fast_path=True`
   - `progressive_step3_skipped=True`
   - `progressive_step2_skipped=True`
   - `questionnaire_summary_skipped=True`
4. 两种模式下 `requirements_analyst_review` 门禁逻辑必须一致（`blocker/warn/pass` 行为不变）。

#### 2.4.13 Fast Path 的“27方法最小保障守卫”（已实施）

为避免“开了最短路径就绕过分析质量”的误解，当前实现新增了 Fast Path 守卫：

1. 守卫方法：`_can_apply_requirements_fast_path(state)`
2. 生效位置：`_unified_input_validator_secondary_node()`
3. 行为：
   - 守卫通过：允许短路到 `project_director`
   - 守卫不通过：自动回退默认链路（继续 step3/step2/summary）

守卫检查项（当前实现）：

1. **审阅门禁必须 `pass`**
   - 读取：`state.requirements_quality_gate.severity`
2. **核心结构化产物必须存在**
   - `project_type`
   - `space_classification`
   - `task_framework`
   - `structured_requirements`
3. **27方法关键字段最小承载（M18/M19/M22/M23/M24）**
   - M18：`structured_requirements.project_motivation_detail`
   - M19：`structured_requirements.design_motivation_detail`
   - M22：`structured_requirements.stance_spectrum.rationale`
   - M23：`structured_requirements.expert_handoff_questions.universal_disruption_challenge`
   - M24：`structured_requirements.disruption_authorization.disruption_rationale`

运行时可观测字段：

- `requirements_fast_path_guard.passed`
- `requirements_fast_path_guard.reasons`

说明：

1. 该守卫确保“最短路径”只跳过二级补全步骤，不绕过需求分析核心质量；
2. 若关键字段缺失，系统会自动退回默认链路做补全；
3. 该策略不改主图拓扑，仅在节点内动态决策。

## 3. 设计目标与非目标

### 3.1 设计目标

1. 保持现有 `RequirementsAnalystAgentV2` 主引擎不废弃。
2. 在不破坏主线骨架的前提下，把“七层结构”逐步嵌入一级大节点 `需求分析` 的内部。
3. 让当前散字段输出开始有明确归属与收口点。
4. 让高复杂项目（如乡村文旅、民宿集群、文化示范项目）能从“深度分析”升级到“可供下游方案继续工作的中间对象”。
5. 为 `需求分析` 节点内部的 review / 校正 / 定稿子链提供稳定对象协议。
6. 为后续 `questionnaire_summary.py`、`project_director.py`、前端审阅页提供更稳定的输入协议。

### 3.2 非目标

1. 第一轮 **不** 把 27 方法做成多 Agent / 多工位执行系统。
2. 第一轮 **不** 完全重写 `RequirementsAnalystAgentV2`。
3. 第一轮 **不** 让“层内风暴”变成运行时复杂机制。
4. 第一轮 **不** 直接追求“自动生成完整建筑/室内方案稿”。
5. 第一轮 **不** 试图一次性冻结第 1-5 层全部对象协议。

### 3.3 开发方法原则（当前最高优先级）

本轮以及后续同类开发，正式采用以下方法原则：

> **每一次问题测试，都是一次加强、反思、查漏补缺、持续提升。**

也就是说，“多轮问题测试”本身就是开发前的核心迭代过程，而不是编码完成后的补救性验证。

#### 原则解释

过去“先开发，再测试，再修改，再反复返工”的方式，在高复杂需求分析系统中效率过低，且容易导致：

1. 架构边界尚未稳定就过早编码。
2. 问题定义本身不完整，导致实现对象反复推倒重来。
3. 测试被动承担架构纠偏职责，进入低效循环。

因此，当前方法必须改为：

1. **先做问题测试与案例模拟。**
2. **在测试中反思边界、发现缺口、补充协议。**
3. **把稳定下来的认知与规则先写入文档。**
4. **最后再进入代码开发。**

#### 对本项目的直接含义

在“需求分析师七层结构”开发中：

1. 每一层都应先进行多轮案例测试，再进入正式实现。
2. 每个新案例都不只是“验证旧结论”，还要承担“加强标准、暴露盲区、修正文档”的职责。
3. 如果测试暴露出对象边界不清、字段协议不足、层级职责混淆，应优先修正文档与原则，而不是立刻写补丁代码。
4. 编码启动条件不再是“已经有一个初步想法”，而是“经过多轮测试后，该层职责、边界、输出骨架已足够稳定”。

#### 执行要求

后续推进时，应默认采用以下顺序：

`问题测试 / 案例模拟 -> 反思与查漏补缺 -> 文档修正 -> 再进入代码实现`

若跳过前述步骤直接编码，应视为偏离当前最高优先级开发原则。

#### 每轮测试的固定交付要求

从当前版本开始，每一次问题测试在交付时都必须显式包含以下三部分：

1. **对系统的启发**
   - 本次测试让系统确认了什么新能力、新边界或新输入类型。
2. **需要加强的部分**
   - 本次测试暴露了哪些字段不足、边界模糊、失败风险或协议缺口。
3. **需要写入开发文档的内容**
   - 本次测试中哪些结论应沉淀为正式规则、字段增强、测试口径或失败样式。

如果一次测试只给出拆解结果，而没有明确说明以上三点，应视为测试交付不完整。

#### 用户确认后的执行要求

当用户对某一条测试结果回复“确认”后，系统不得只停留在展示层确认，而必须立即执行以下动作：

1. 将该条测试中确认成立的增强点、边界修正、失败样式或测试口径写入开发文档。
2. 将必要的冻结结论同步到仓库记忆，避免后续轮次遗失。
3. 完成上述写入动作后，方可进入下一条问题测试。

也就是说：

> **用户确认 = 必须发生真实沉淀动作，而不只是口头确认。**

#### 自动确认授权（2026-03-07 新增冻结）

自 2026-03-07 起，用户已明确授权：

> **后续问题的第 1 层测试，默认自动确认。**

这意味着：

1. 后续 `sf/q.txt` 顺序问题在完成第 1 层测试交付后，无需再等待逐条人工回复“确认”。
2. 系统应将该次第 1 层测试视为已获确认，并立即执行文档沉淀与仓库记忆写入。
3. 完成沉淀后，可直接进入下一条问题的第 1 层测试。

注意：

- 此授权当前仅适用于**后续问题的第 1 层测试流程**；
- 不等于跳过“沉淀动作”；
- 不等于对未来代码实现、对象冻结或跨层结论的一次性默认批准。

## 5. 七层结构在需求分析师内部的落地方式

### 5.1 层级落地原则

七层结构不是给工作流新增七个运行时节点，而是：

- 一级大节点 `需求分析` 内部的认知装配线
- 先作为需求分析师内部的认知组织方式
- 再逐步变成显式对象与服务拆分方式

因此需要始终区分三件事：

1. **一级大节点**：面向用户理解的主线骨架；
2. **二级步骤**：`需求分析` 节点内部对外可感知的推进环节，如初次呈现、意图确认、信息补全、需求洞察；
3. **七层结构**：`requirements_analyst` 等内部源头能力实际如何逐层看清问题、形成 `TaskFramework` 的内部装配线。

七层不能重新长成七个用户步骤，也不能与一级大节点并列陈述。

### 5.2 当前建议的内部落地顺序

下面的“第 1-7 层”描述的不是用户看到的页面顺序，而是一级大节点 `需求分析` 内部从分析风暴到对象收口的实现顺序；其产物随后会被 `requirements_analyst_review`、`output_intent_detection`、信息补全、可选偏好雷达与 `questionnaire_summary` 继续消费。

#### 第 1 层：输入拆解层
- 责任：把用户原始输入拆成问题像素
- 当前主要落点：`precheck_node` + `phase1_node` 前段
- 本轮输出目标：`ProblemPixels`

#### 第 2 层：识别归类层
- 责任：识别项目/空间类型、变化意图、空间区域
- 当前主要落点：`_type_registry.py`、`project_type_detector.py`、`change_intent_detector.py`
- 本轮输出目标：`SpaceClassification`

##### 第 2 层实施定位（当前冻结口径）

第 2 层当前正式定义为：

> **问题结构的语义归类层 / 桥接分类层**

它的职责不是生成方案，也不是决定读取哪套分析框架，而是把第 1 层 `ProblemPixels` 收束成一个可供第 3 层及之后继续消费的、带解释依据的分类对象。

换句话说，第 2 层要回答的是：

1. 这到底更接近哪一类项目 / 空间问题；
2. 当前输入是否带有明确的改造 / 变化意图；
3. 当前问题落在哪些空间区域、空间单元或组织尺度上；
4. 系统为什么做出这样的归类，而不是只给出一个黑箱标签。

##### 第 2 层明确做什么

1. 识别主项目类型 / 空间类型。
2. 提供次选类型与歧义提示，而不是只给单一标签。
3. 在存在改造语境时识别 `change_intent`。
4. 在有足够依据时提取 `spatial_zones` 或空间单元线索。
5. 输出带解释依据的 `SpaceClassification`，供第 3 层框架调用与第 7 层审阅消费。

##### 第 2 层明确不做什么

1. 不决定读取哪套分析框架（这是第 3 层）。
2. 不把建筑师参考、品牌参考直接翻译成风格答案。
3. 不输出概念方案、材料策略或空间组织结论。
4. 不把 `Mode Engine`、`ontology` 或审阅节点的后置判断混入第 2 层真相源。

##### 第 2 层现有资产清单与能力评级

基于当前真实代码资产审计，第 2 层并不是“没有底盘”，而是“底盘已存在，但对象未收口、主链接线未完成”。当前资产与评级如下：

| 资产类别 | 真实落点 | 当前能力 | 评级 | 结论 |
| --- | --- | --- | --- | --- |
| 项目 / 空间类型真相源 | `services/_type_registry.py` | 已维护 40+ 类型、优先级、标签、关键词、扩展机制 | A | 可直接作为第 2 层唯一真相源 |
| 项目 / 空间类型识别算法 | `services/project_type_detector.py` | 已具备主次关键词、权重、排他性、否定过滤、歧义提示 | A | 足够支撑第 2 层最小版主分类 |
| 变更意图识别 | `services/change_intent_detector.py` | 三分类已成熟：`VISUAL_ONLY` / `SPATIAL_ZONE` / `IDENTITY_PATTERN` | B+ | 算法够用，但主链接线不足 |
| 空间区域 / 空间单元提取 | `interaction/nodes/_oid_constraint.py`、`output_intent_detection.py` | 已有规则提取能力，但位置偏后、来源不稳定 | B- | 可复用，但不能直接当第 2 层正式生产链 |
| 主链承载与写回 | `agents/requirements_analyst_agent.py`、`workflow/nodes/requirements_nodes.py` | 已写回 `project_type`，未写回 `space_classification` | C+ | 这是当前第 2 层最大缺口 |

##### 第 2 层当前是否“够用”

结论分两层：

1. **对“最小版实施”来说，现有资产已经够用。**
   - 已有类型识别真相源；
   - 已有主分类算法；
   - 已有可借用的变化意图与空间区域资产。
2. **对“完整成熟版”来说，现阶段仍然不够。**
   - 还没有正式的 `SpaceClassification` 对象；
   - `spatial_zones` 没有稳定生产者；
   - `change_intent` 未正式并入需求分析主链；
   - 审阅与下游节点尚未以该对象作为优先输入。

因此，第 2 层本轮开发重点不应是“再发明一套新算法”，而应是：

> **把现有分类能力收束成单一对象，并稳定接到主链。**

##### `SpaceClassification` 最小字段（当前建议冻结版）

第 2 层当前建议先冻结以下最小字段集，作为桥接草案对象的第一版协议：

- `primary_type`
- `primary_type_label`
- `type_confidence`
- `candidate_types`
- `spatial_zones`
- `change_intent`
- `parent_type`
- `specialty_tag`
- `detection_method`
- `detection_signals`

其中字段边界规定如下：

1. `primary_type`
   - 来自 `PROJECT_TYPE_REGISTRY` 的类型 ID；
   - 是第 2 层的主分类结果。
2. `primary_type_label`
   - 来自注册表标签映射；
   - 仅用于可读性与审阅展示。
3. `type_confidence`
   - 当前允许采用规则匹配得分归一化；
   - 第一轮不要求引入更复杂概率模型。
4. `candidate_types`
   - 用于保留次选类型与歧义提示；
   - 防止第 2 层过早写死单一结论。
5. `spatial_zones`
   - 当前允许为空；
   - 有稳定依据时再由规则提取结果填充。
6. `change_intent`
   - 当前允许为空；
   - 仅在存在修改 / 改造语境时填充。
7. `parent_type` / `specialty_tag`
   - 用于把主类型进一步映射到更宽泛业态与专项标签；
   - 便于第 3 层框架调用与审阅解释。
8. `detection_method` / `detection_signals`
   - 用于保留分类依据；
   - 是第 2 层“可解释输出”的必要组成，不是调试噪音。

##### 第 2 层当前实施顺序（P0 / P1）

为避免第 2 层再次滑成“大而全重构”，当前建议按以下顺序推进：

1. **P0：先把 `SpaceClassification` 对象正式产出并写回主链。**
   - 至少收束 `primary_type`、`primary_type_label`、`type_confidence`、`candidate_types`、`detection_signals`；
   - 保持现有 `project_type` 历史字段兼容输出。
2. **P1：补入 `spatial_zones` 的稳定生产逻辑。**
   - 可先复用现有规则提取，不追求一步到位；
   - 但必须明确来源与写回位置。

当前实现状态（2026-03-07）：

- 第 2 层已不再单独维护一套临时 `spatial_zones` 提取规则；
- 已优先复用 `interaction/nodes/_oid_constraint.py::_extract_spatial_zones()` 作为共享提取入口；
- 再由需求分析师内部做字段归一化，写回 `space_classification.spatial_zones` 与 `extracted_spatial_zones`；
- 这意味着第 2 层与后置 OID 链路已开始共用同一条空间区域提取底盘。

3. **P1：在修改 / 改造语境下接入 `change_intent`。**
   - 优先服务明确改造场景；
   - 新创建场景可暂时允许为空。
4. **P1：让审阅层优先显示 `SpaceClassification` 摘要。**
   - 但必须保留旧字段回退，不得破坏现有 review 与回归测试。

当前实现状态（2026-03-07，review 适配）：

- `interaction/nodes/requirements_analyst_review.py::_build_review_payload()` 已开始优先读取 `space_classification`、`project_classification`、`extracted_spatial_zones`；
- review payload 已新增：
   - `space_classification`
   - `project_classification`
   - `extracted_spatial_zones`
   - `space_classification_summary`
- `space_classification_summary` 当前最小职责是向审阅层提供：主类型、置信度、候选类型、变化意图、空间区域焦点与简短解释文本；
- 若 `structured_requirements` 未携带该对象，审阅层仍允许从 state 顶层字段回退读取；
- `tests/regression/test_requirements_review_trace_regression.py` 已补充回归测试并通过，确认 review 层消费第 2 层对象不会破坏现有机制线与门禁逻辑。

##### 第 2 层当前阶段性结论

截至当前，关于第 2 层可以先冻结以下共识：

1. 第 2 层的核心问题不是“缺算法”，而是“缺显式对象与接线”。
2. `PROJECT_TYPE_REGISTRY` + `ProjectTypeDetector` 已足以支撑第 2 层最小可用版。
3. `change_intent_detector.py` 与空间区域提取逻辑属于可复用资产，但当前还不是稳定主链产出。
4. 第 2 层本轮的正确目标是：
   - 先收口为 `SpaceClassification`；
   - 再让第 3 层、第 7 层开始稳定消费；
   - 而不是一开始就追求完整智能分类宇宙。 

#### 第 3 层：框架调用层
- 责任：按类型、模式、必要 ontology 片段读取分析骨架
- 当前主要落点：`space_type_dimension_map.yaml`、`sf/10_Mode_Engine`、`mode_detector.py`、`ontology_loader.py`

### 5.3 第 1 层输入拆解层：专项模拟与判定样例

> 用途：本节不是代码设计，而是第 1 层“输入拆解层”的标准测试样例。后续开发、联调、评审都应先对照本节，确认系统是否真的只做了“拆解”，而没有越级进入分类、风格判断或方案生成。

#### 测试输入

`四川广元苍溪云峰镇狮岭村进行新农村建设升级，计划打造具有文化示范意义的民宿集群。要求深度挖掘在地农耕文化、产业结构与乡村经济逻辑，同时融合安藤忠雄的精神性空间、隈研吾的材料诗性，以及刘家琨、王澍、谢柯等中国建筑师的地域智慧，形成商业成功与文化深度兼具的项目模型。请提出整体规划与室内设计概念框架，包括建筑语言、公共空间组织、材料策略与运营模型。`

#### 第 1 层允许做什么

输入拆解层在本例中**只允许完成以下工作**：

1. 识别地点、对象、动作、目标、显性交付请求。
2. 识别用户原文中明确出现的文化/方法参考源。
3. 识别任务里的多目标并列关系。
4. 标记潜在缺口、模糊项与后续必须确认的信息。
5. 把原始长 brief 拆成可供后续层消费的问题像素。

#### 第 1 层明确不做什么

在这一层，**禁止**提前做以下事情：

1. 禁止直接给出项目类型最终定性（这属于第 2 层）。
2. 禁止判断应读取哪套空间框架（这属于第 3 层）。
3. 禁止把建筑师参考直接解释成风格方案（这属于第 3/6 层之后）。
4. 禁止提前输出完整概念方案、总平策略或室内设计答案。
5. 禁止把推断当成事实写死。

#### 本例的标准拆解结果（`ProblemPixels` 草案）

以下结果是当前需求分析师第 1 层应尽量达到的“合格拆解”，不是后续层的完整分析结果。

##### A. 地点与场地像素

- 省市县镇村：四川 / 广元 / 苍溪 / 云峰镇 / 狮岭村
- 场景背景：新农村建设升级
- 空间发生地：乡村聚落环境
- 潜在场地属性：村庄更新、乡建、文旅导入场景

##### B. 项目动作像素

- 动作 1：进行新农村建设升级
- 动作 2：打造民宿集群
- 动作 3：建立文化示范意义
- 动作 4：形成可兼顾商业与文化深度的项目模型

##### C. 显性目标像素

- 目标 1：深挖在地农耕文化
- 目标 2：理解产业结构与乡村经济逻辑
- 目标 3：形成整体规划与室内设计概念框架

#### 本例的阶段验收标准

1. 一上来就输出“这是高端乡村文旅度假项目，应采用清水混凝土+木格栅”。
2. 直接把安藤 / 隈研吾 / 中国建筑师语言写成方案结论。
3. 完全不标记预算、运营、审批、规模等缺口。
4. 只提炼出一句空泛摘要，没有形成结构化问题像素。
5. 把第 2 层分类工作、第 3 层框架读取工作提前做掉。

### 5.4 第 1 层统一深度标准（当前冻结口径）

基于前述样例与后续对话测试结果，第 1 层现统一定义为：

> **高分辨率结构化拆解层**，而不是“输入摘要层”。

也就是说，第 1 层不是把用户原话压缩成一句话，而是要把原始 brief 中能够被后续层继续消费的结构化问题像素，尽可能完整、清楚、分边界地拆出来。

#### 统一要求

1. 第 1 层输出深度，必须至少达到当前案例模拟的粒度。
2. 第 1 层必须“足够深”，但仍然只能停留在“拆解”边界内。
3. 第 1 层既要能处理“信息稀疏但目标复杂”的输入，也要能处理“信息密集且关系复杂”的输入。
4. 第 1 层的合格标准，不是摘要是否顺口，而是是否为第 2 层及之后提供了稳定、可继承的像素底板。

#### 推荐统一输出骨架（`ProblemPixels` 当前建议版）

在当前阶段，第 1 层建议至少尝试输出以下 10 组像素：

- `context_pixels`
- `fact_pixels`
- `program_pixels`
- `goal_pixels`
- `deliverable_pixels`
- `reference_pixels`
- `stakeholder_hint_pixels`
- `tension_signal_pixels`
- `gap_pixels`
- `risk_hint_pixels`

#### 第 1 层主骨架稳定性结论（基于 `sf/q.txt` 题库校准）

基于当前 `sf/q.txt` 全量问题题库的持续校准，第 1 层主骨架现正式确认如下：

1. 上述 10 组 `ProblemPixels` 继续作为第 1 层**稳定主骨架**。
2. 当前不因题型扩张而新增 `constraint_pixels`、`task_route_pixels` 等主字段。
3. 当前优先通过以下方式吸收边界压力，而不是继续增加主字段数量：
   - 扩展现有字段语义；
   - 增加辅助标签层；
   - 增加按题型的输出规则与命中标准。
4. 第 1 层主字段按“信息的认识论角色”分类，而不是按“问题领域”分类。

这意味着：

- 不因出现心理题、方法论题、极端环境题、规划题，就按领域拆出新的主字段；
- 第 1 层要优先保持主骨架稳定，再通过字段内语义增强来提升适配力；
- 若未来确有新的输入类型对主骨架形成长期、稳定、不可吸收的结构性压力，必须先经过连续案例验证与文档修正，再讨论是否扩展主字段。

#### 当前不升为主字段的两个高频提醒项

基于近期交叉评估，以下两类信息虽然重要，但当前**不单独升为第 1 层主字段**：

1. `constraint_pixels`
   - 原因：硬约束、软约束、运营约束分别属于事实、缺口、风险提示等不同职责位；
   - 若单独新增，容易导致同一约束被重复记录于多个字段中，破坏字段边界。
2. `task_route_pixels`
   - 原因：任务路由本质上属于 `program_pixels` 的任务程序子语义；
   - 若单独新增，容易使第 1 层过早滑入流程设计或运行时路由判断。

因此，当前阶段采取的口径是：

> **重要，不等于必须升为主字段。**

只要某类信息仍能被现有字段通过明确规则稳定承接，就优先保留 10 组主骨架不变。

说明：

- `context_pixels`：地点、项目背景、场地/组织语境、任务发生场。
- `fact_pixels`：用户文本中明确给出的硬事实，不允许混入判断。
- `program_pixels`：空间程序、功能分区、尺度对象、使用场景。
- `goal_pixels`：显性目标、隐含追求、任务导向。
- `deliverable_pixels`：用户明确要求产出的内容或聚焦部位。
- `reference_pixels`：人物、方法、品牌、案例、审美线索等参考输入。
- `stakeholder_hint_pixels`：可以提示但尚未展开成系统分析的角色线索。
- `tension_signal_pixels`：已显现的冲突、拉扯、双重目标或尺度错位。
- `gap_pixels`：当前输入没有给出、但后续推进必需的信息缺口。
- `risk_hint_pixels`：当前 brief 中已能预警的误读风险、退化风险、执行风险。

#### 第 1 层统一输出规则（当前冻结口径）

1. 第 1 层必须**扫描全部 10 组字段**。
2. 第 1 层必须返回完整的 10 组字段 key。
3. 但第 1 层**不要求每组字段都必须非空**。
4. 若某组字段在当前输入中确实缺乏依据，应输出空列表，而不是为了追求完整率而补写。
5. “空字段”在第 1 层是允许的；“无依据补写”应视为失败样式。

也就是说：

> **第 1 层要求的是“完整扫描”，不是“强行填满”。**

#### 5.4A 第 1 层复盘与简化原则（2026-03-08 新增冻结）

基于近期连续测试与交叉反思，第 1 层当前确认如下：

> **问题不在 10 字段主骨架，而在并行解释层过多，正在把“拆解层”写成“高解释密度观察层”。**

因此，第 1 层当前正式采用以下简化口径：

1. **保留 10 组 `ProblemPixels` 主骨架不变。**
   - 当前骨架已能覆盖项目型、人物型、研究型、技术系统型、规划治理型输入；
   - 不再因为题型扩张继续增加主字段数量。
2. **第 1 层不是创意生成层，只允许有限开放。**
   - 允许 LLM 在 `stakeholder_hint_pixels`、`tension_signal_pixels`、`risk_hint_pixels`、`open_signal_pixels` 中做受控补漏；
   - 允许显式记录 `ambiguity_flags` 与 `global_ambiguity_flags`；
   - 不允许 LLM 以自由发挥方式改写 `fact_pixels`、提前生成类型结论、框架结论或方案结论。
3. **第 1 层优先采用“规则优先 + 轻量语义补漏 + 统一元标记”的最小组合。**
   - 规则负责保底；
   - 轻量语义补漏负责发现未被硬规则吸收、但值得保留的线索；
   - 统一元标记负责标出事实 / 线索 / 缺口 / 风险 / 模糊 / 受控推断；
   - 当前不引入多套并列影子对象来解释同一轮拆解。
4. **第 1 层的目标是为第 2 层提供稳定底板，而不是在本层完成“解释闭环”。**
   - 若某些语义尚无法稳定收束，应优先保留为空字段、模糊标记或开放信号；
   - 不为追求“看起来很懂”而把未定义概念写成稳定前提。

#### 第 1 层最小诊断层（`problem_pixel_diagnostics`）

在保留 `ProblemPixels` 主骨架的前提下，第 1 层当前仅保留一个**最小诊断对象**：`problem_pixel_diagnostics`。

它的职责只有三类：

1. 说明当前 10 组字段的命中状态是否稳定；
2. 标记当前输入中的模糊区、开放区与低置信区域；
3. 为第 2 层及审阅层提供“哪些地方不要过度相信”的提醒。

当前最小建议结构如下：

- `observer_mode`
   - 当前固定为 `protocol_observer`，强调它仍然是协议观察补充层，而不是第二主结论层。
- `field_diagnostics`
   - 按 10 组 `ProblemPixels` 字段返回最小状态：`status`、`confidence`、`source_types`、`ambiguity_flags`、`notes`。
- `open_signal_pixels`
   - 暂未被稳定协议吸收、但值得后续继续消费的开放信号。
- `global_ambiguity_flags`
   - 跨字段级别的整体模糊提示。

冻结边界：

1. `problem_pixel_diagnostics` 是**唯一保留的第 1 层并行对象**，不是新的主骨架字段集合。
2. 它只能补充解释，不得生成与 `problem_pixels` 平级竞争的第二套字段骨架。
3. 当前 P0 阶段不再把 `problem_pixel_shadow_analysis`、`problem_pixel_llm_shadow_alignment` 作为第 1 层必备合同对象。
4. 若未来确实需要更自由的盲区探针，应先证明最小诊断层不足，再以“调试视图”而非“主链必备对象”方式增补。
5. 诊断层的存在是为了提醒边界，不是为了让第 1 层承担更多解释责任。

#### 第 1 层下游消费摘要（`problem_pixel_observability_briefing`）

为避免下游直接散读诊断层，当前仍保留一个**轻量消费摘要**：`problem_pixel_observability_briefing`。

但该对象当前只允许压缩以下内容：

1. `problem_pixel_diagnostics` 的开放信号；
2. `problem_pixel_diagnostics` 的全局模糊标记；
3. 当前最值得下游继续验证的 1~3 个优先提醒。

当前主链约定如下：

1. `questionnaire_summary.py` 若消费第 1 层诊断信息，应优先压缩为 `problem_pixel_observability_briefing` 或其轻量文本摘要，而不是散拼原始诊断字段。
2. `project_director.py` 若需要关注第 1 层残余不确定性，应优先读取该摘要，而不是直接读取完整诊断对象。
3. 该摘要只能表达“待验证提醒”，不得把开放信号或模糊标记升级为新的事实真相层。
4. 若当前输入不存在显著开放信号或模糊区，该摘要允许为空。
5. 当前最小结构应至少包含：`mode`、`summary_text`、`priority_alerts`、`downstream_focus`、`recommendation`。

#### 当前明确下调为“非 P0 必做”的设计

以下设计当前不再作为第 1 层的必做实现要求：

1. `problem_pixel_shadow_analysis`
2. `problem_pixel_llm_shadow_alignment`
3. 面向第 1 层的多轮并行对照摘要链
4. 为同一轮拆解同时维护多份 coverage / blind spot / alignment 对照对象

原因不是这些想法永远无价值，而是它们在当前阶段会带来以下问题：

1. 让第 1 层的实现复杂度高于其职责复杂度；
2. 让开发重心从“稳定拆解”偏移到“重复解释拆解”；
3. 让下游消费端面对过多中间对象，增加理解与接线成本；
4. 容易诱导团队把第 1 层误当作自由分析层，而非严格边界层。

#### 按题型的最低命中原则（当前建议口径）

为避免用统一标准误伤不同题型，第 1 层当前建议采用“按题型最低命中”而不是“10/10 全非空”的口径。

1. **项目 / 空间型输入**
   - 优先命中：`context_pixels`、`program_pixels`、`goal_pixels`、`deliverable_pixels`、`gap_pixels`
2. **人物 / 精神驱动型输入**
   - 优先命中：`reference_pixels`、`stakeholder_hint_pixels`、`tension_signal_pixels`、`goal_pixels`、`gap_pixels`
3. **研究 / 方法论型输入**
   - 优先命中：`context_pixels`、`program_pixels`、`goal_pixels`、`deliverable_pixels`、`reference_pixels`、`gap_pixels`
4. **技术 / 极端环境型输入**
   - 优先命中：`fact_pixels`、`context_pixels`、`program_pixels`、`deliverable_pixels`、`gap_pixels`、`risk_hint_pixels`
5. **规划 / 更新 / 治理型输入**
   - 优先命中：`context_pixels`、`goal_pixels`、`deliverable_pixels`、`stakeholder_hint_pixels`、`tension_signal_pixels`、`gap_pixels`

上述原则的作用不是给字段“打分”，而是提醒第 1 层：

- 不同问题有不同的高价值识别区；
- 不应拿住宅题的字段密度去要求研究题；
- 不应拿研究题的稀疏结构去要求极端技术题。

#### 事实、线索、缺口、推断的边界规则

第 1 层必须明确区分以下四类信息：

1. **事实（Fact）**：用户原文直接给出的内容。
2. **线索（Hint / Signal）**：由原文强相关导出的提示，但尚不能写成结论。
3. **缺口（Gap）**：推进任务所必需但当前未提供的信息。
4. **受控推断（Controlled Inference）**：仅允许在极少数必要场景下，以显式标签如 `[推断]` 标记，并且必须能回溯到文本来源。

#### 第 1 层允许的推断

- 允许把“某类任务天然涉及的角色”标为 `stakeholder_hint_pixels`。
- 允许把“多目标之间的拉扯”标为 `tension_signal_pixels`。
- 允许把“显著的误读风险”标为 `risk_hint_pixels`。

#### 第 1 层禁止的推断

- 禁止把项目直接定型为某个最终空间类别或项目类别。
- 禁止把参考人物 / 方法直接翻译成风格答案。
- 禁止把尚未给出的预算、面积、客群、运营方式、审批条件写成事实。
- 禁止把用户的气质线索直接写成完整人物设定结论。
- 禁止提前生成概念方案、设计策略或材料答案。

#### 阶段验收补充标准

如果第 1 层实现正确，那么它除了能完成前述专项样例，还应满足：

1. 输出结果至少形成 10 组稳定像素分区，而不是散点摘录。
2. 每组像素都能说明它属于事实、线索、缺口还是风险提示。
3. 即使输入高度复杂，第 1 层也不得越级做第 2 层分类结论。
4. 即使输入带有强烈设计暗示，第 1 层也不得越级给出设计答案。
5. 下游读取者无需回看原始长 brief，也能看懂“问题是如何被拆开的”。

### 5.5 第 1 层补充测试口径（新增）

除“乡村文化示范型民宿集群”外，第 1 层还必须通过另一类校准测试：

- **高密度复合型输入**：例如“总部大楼中的旗舰高级公寓复合项目”。

这一类输入验证的不是“文化叙事”，而是第 1 层是否能在信息很多、关系很密的情况下，仍然把以下内容拆开：

1. 建筑背景与城市语境。
2. 空间程序与楼层功能分布。
3. 品牌目标、商业目标与体验目标。
4. 方法参考与显性交付请求。
5. 张力信号、约束线索与信息缺口。

因此，第 1 层后续开发与测试不能只围绕“模糊复杂输入”展开，也必须覆盖“高密度复合输入”。

### 5.6 第 1 层适用问题类型补充（基于多轮测试）

基于当前多轮问题测试结果，第 1 层“输入拆解层”不得再被理解为只服务于“项目 brief 拆解”。

当前至少应覆盖以下三类输入：

1. **项目型输入**
   - 典型表现：地点、规模、功能、交付请求明确。
   - 例如：乡村民宿集群、总部中的旗舰公寓、海景别墅等。
2. **人物 / 场景驱动型输入**
   - 典型表现：人物气质、身份结构、生活方式、重点空间与表达诉求共同构成 brief。
   - 例如：以某类业主原型、某类人物精神结构、某类生活场景为出发点的设计问题。
3. **研究 / 方法论型输入**
   - 典型表现：没有具体项目参数，但存在明确的分析对象、变量关系、方法论诉求与应用转译要求。
   - 例如：文化变量、社会进程变量、设计概念结构之间关系的系统分析问题。

这意味着第 1 层的职责不是“拆项目说明书”，而是：

> **拆一切可被后续分析链继续消费的问题输入。**

### 5.7 第 1 层字段语义增强（基于多轮测试）

在保持当前 10 组 `ProblemPixels` 骨架不变的前提下，基于新案例测试，现补充以下字段语义增强要求。

#### 5.7.1 `program_pixels` 不得被狭义理解为“空间功能表”

`program_pixels` 当前应支持至少两种形态：

1. **空间程序**
   - 如功能分区、楼层关系、公共/私密区域、关键界面空间。
2. **任务程序**
   - 如先分析什么、再转译什么、最终产出什么的结构化处理单元。

3. **系统 / 运营程序**
   - 如施工 / 装配 / 拆卸序列、运营流程、数据反馈回路、技术系统组织单元。

也就是说，`program_pixels` 更准确的理解应是：

> **任务中的结构化处理单元**

而不仅仅是房间清单。

补充边界规则：

1. 允许记录 3 级以内的关键任务序列或系统程序牵引点；
2. 不要求在第 1 层展开详细流程图、施工工艺图或技术路线图；
3. 第 1 层只负责识别“有哪些程序单元存在”，不负责设计完整实施流程。

#### 5.7.2A `context_pixels` 需显式覆盖物理环境与系统语境

基于 `sf/q.txt` 后续题库扩展，第 1 层不得再把 `context_pixels` 狭义理解为“地点 + 背景介绍”。

当前 `context_pixels` 至少还应支持以下内容：

1. **场地物理条件**
   - 如高海拔、盐雾、极寒、强紫外线、潮湿、防空洞、地下空间等。
2. **制度 / 治理语境**
   - 如政府主导、保障房、公共示范、应急避难、非营利组织等。
3. **产业 / 运营语境**
   - 如竞标、空置率、坪效、夜经济、招商、品牌升级、长期运营闭环等。
4. **跨文化 / 地缘语境**
   - 如跨国家庭、一带一路、多语境身份流动、本土扎根与全球视野并存等。

也就是说，`context_pixels` 当前更准确的理解应是：

> **问题成立所依赖的外部语境与环境条件。**

#### 5.7.3 `gap_pixels` 必须同时覆盖物理缺口与语义缺口

#### 5.7.2 `reference_pixels` 必须支持更广义的参考输入

当前 `reference_pixels` 不仅包括：

- 建筑师
- 设计师
- 品牌
- 项目参考
- 方法线索

还必须支持：

- 文化变量输入
- 社会进程变量输入
- 人物精神原型输入
- 身份结构输入
- 表达边界输入
- 场景资源输入

说明：

- “农耕文化”“城市化进程”属于变量型参考输入；
- “一代创业者的精神结构”属于人物精神原型输入；
- “避免过度炫耀”属于表达边界输入；
- “海景资源”属于场景资源输入。

第 1 层只能记录这些输入存在，不得把它们直接翻译成风格或方案结论。

#### 5.7.3 `gap_pixels` 必须同时覆盖物理缺口与语义缺口

当前第 1 层不得只会识别预算、面积、工期、客群这类硬信息缺口。

还必须识别：

1. **物理缺口**
   - 面积、尺度、预算、工期、结构条件、功能分布等。
2. **语义缺口**
   - 某个关键概念虽然被提出，但其真实含义尚未定义清楚。
3. **评价标准缺口**
   - 某个目标词被提出，但其成功判据、评价标准或落地尺度尚未定义清楚。

例如：

- “家族传承”到底指家风、记忆、秩序还是资产象征；
- “内心回归”到底指休息感、去表演化还是精神归属；
- “厚度的精神表达”到底是审美克制、价值沉淀还是叙事纵深。

如果系统不能识别这种语义未定义状态，第 1 层就会过早把模糊概念写死为后续方案前提。

其中，评价标准缺口的典型例子包括：

- “高端感”的判断标准是什么；
- “示范意义”的衡量方式是什么；
- “商业成功”的评价周期与阈值是什么；
- “有尊严”的公共或居住标准是什么。

如果系统不能识别这种“目标已提出、判据未定义”的状态，就会把口号直接误当作稳定前提。

#### 5.7.3A `fact_pixels` 需显式支持“硬约束 / 实施事实”

当前 `fact_pixels` 不得只理解为“面积、预算、时间”之类的数字事实。

还必须支持：

1. **硬约束事实**
   - 如不得闭店、不得大拆大建、预算上限、工期上限、能源或安全等级下限。
2. **技术事实**
   - 如医疗级新风、恒温恒湿、供氧系统、隔振体系、低敏材料等已被明确提出的技术边界。
3. **实施事实**
   - 如夜间施工、可拆装回收、连续运营、7×24 小时运转等已被明确提出的执行条件。

这意味着：

> **硬约束首先是事实，而不是风险。**

第 1 层必须先把其识别为事实或缺口，再去判断其与目标之间可能形成的风险。

#### 5.7.4 `tension_signal_pixels` 需支持“身份—精神—空间”复合张力

在高复杂住宅、人物驱动型任务或方法论型任务中，张力往往不是简单的“双目标冲突”，而是复合结构。

第 1 层至少要能标记如下类型的张力：

- 身份表达 vs 内在回归
- 公共权力结构 vs 私密精神安放
- 文化根性 vs 当代转译
- 抽象方法论 vs 实际应用路径
- 外在资源展示性 vs 内在空间厚度

第 1 层不裁决这些张力，只负责把它们显式化，为第 4-6 层保留判断基础。

#### 5.7.5 第 1 层需显式支持“人物精神原型”输入

经多轮测试确认，第 1 层必须把如下输入类型视为可正式识别对象：

- 创业者精神结构
- 年轻业主人物原型
- 家族角色关系
- 身份跃迁叙事
- 气质与表达边界的复合输入

注意：

第 1 层只允许将其记录为：

- `reference_pixels`
- `stakeholder_hint_pixels`
- `tension_signal_pixels`
- `gap_pixels`

不得直接把人物原型翻译为完整空间人格或完整设计概念。

#### 5.7.6 第 1 层需支持“品牌升级 / 竞争驱动型输入”

经 `sf/q.txt` 第 1 条测试确认，第 1 层除处理项目、人物、研究型输入外，还必须稳定处理：

- 品牌升级型输入
- 旗舰化目标输入
- 竞争驱动型输入
- 价值感知型输入

典型表现包括：

- 品牌正处于产品迭代或升级阶段；
- 项目承担旗舰店、示范店、样板型门店职责；
- 任务中明确存在竞争参照对象；
- brief 中出现“高端感”“超值感”“差异化优势”等价值感知词。

对这类输入，第 1 层至少要能拆出以下内容：

1. **品牌阶段输入**
   - 如升级、迭代、旗舰化、示范化。
2. **竞争关系输入**
   - 如对标对象、替代关系、差异化目标。
3. **价值感知输入**
   - 如“超值感”“豪宅精神”“高端感”“行业示范意义”等。

注意：

- 第 1 层只能将上述内容记录为 `reference_pixels`、`goal_pixels`、`tension_signal_pixels` 与 `gap_pixels` 的组成部分；
- 不得直接把“品牌升级”翻译为空间风格答案；
- 不得直接把“竞争优势”翻译为设计策略答案；
- 不得把“超值感”“豪宅精神”“示范意义”当成已定义清楚的事实。

#### 5.7.7 第 1 层需识别“价值语义缺口”

在品牌升级型、高端项目型或复合商业型输入中，系统需特别识别一类语义缺口：

> **价值语义缺口**

即：某个价值判断词被明确提出，但其真实含义、判断标准或转译方式尚未被定义清楚。

典型例子包括：

- “超值感”
- “豪宅精神”
- “行业示范意义”
- “高端感”
- “生活方式创新”

如果系统不能识别这类缺口，就会过早把价值判断词误当成稳定设计前提，从而导致后续层级滑入空泛概念或误判。

#### 5.7.7A `deliverable_pixels` 需支持设计型之外的交付结构

随着题库扩展，第 1 层不得再把 `deliverable_pixels` 只理解为“设计概念 / 空间框架”。

当前至少还应支持：

1. **研究型交付**
   - 如研究框架、变量关系模型、方法论结构、分析路径。
2. **策略型交付**
   - 如竞标框架、升级策略、更新策略、空间产品策略。
3. **标准型交付**
   - 如保障房标准、可升级模块标准、公共空间标准化框架。
4. **转化型交付**
   - 如品牌转译路径、技术体系转译模型、运营转化路径。

也就是说，`deliverable_pixels` 当前更准确的理解应是：

> **用户要求系统最终产出的结构类型。**

#### 5.7.8 第 1 层需支持“身份跃迁 / 资源—精神联动”输入

经 `sf/q.txt` 第 2 条测试确认，第 1 层还必须稳定识别一类高复杂住宅输入：

- **身份跃迁型输入**
- **资源—精神联动型输入**
- **成就象征—内心回归并存型输入**

典型表现包括：

- 业主存在从草根奋斗到资本成功的明确生命历程；
- 空间期待同时包含外在成就表达与内在精神安放；
- 稀缺景观、豪宅区位、材质象征等资源条件，与精神表达任务同时出现；
- brief 中明确要求避免浅层炫耀，而转向更具厚度的表达。

对这类输入，第 1 层至少要能拆出以下内容：

1. **身份跃迁信号**
   - 如草根奋斗、代际跨越、资本成功、第一代创业者等。
2. **资源条件信号**
   - 如一线海景、别墅区位、材质象征系统、稀缺景观资源等。
3. **精神回归信号**
   - 如内心回归、精神厚度、去炫耀化、沉淀感、家族传承等。

注意：

- 第 1 层只能将上述内容记录为 `reference_pixels`、`tension_signal_pixels`、`gap_pixels` 与 `risk_hint_pixels` 的组成部分；
- 不得直接把“成功人士”翻译成豪宅风格答案；
- 不得直接把海景资源翻译成展示性空间策略；
- 不得把“精神厚度”“内心回归”误当作已经定义清楚的稳定事实。

#### 5.7.9 第 1 层需显式识别“家族系统线索”

经 `sf/q.txt` 第 2 条测试确认，第 1 层在人物驱动型住宅输入中，不能只识别个人气质，还必须显式识别：

- 家族角色关系
- 代际传承诉求
- 公共 / 私密权力关系
- 家庭内部的象征秩序线索

这类内容当前至少应优先进入：

- `stakeholder_hint_pixels`
- `tension_signal_pixels`
- `gap_pixels`

说明：

- “家族传承”不能只被当作情绪词；
- “公共与私密区域权力关系”不能只被当作空间分区要求；
- 它们首先是后续第 4-5 层要继续展开的人本与判断前置信号。

#### 5.7.10 第 1 层需支持“聚落级 / 集群级项目输入”

经 `sf/q.txt` 第 3 条测试确认，第 1 层不能只把问题理解为单体建筑或单空间任务，还必须稳定处理：

- **聚落级输入**
- **集群级输入**
- **村庄更新 / 文旅导入复合输入**

典型表现包括：

- 输入对象不是单体，而是村落、片区、集群或多建筑组织；
- brief 同时要求整体规划与室内概念框架；
- 任务天然横跨场地、建筑、公共空间、运营模型多个尺度。

对这类输入，第 1 层至少要能拆出以下内容：

1. **聚落尺度信号**
   - 如村庄更新、集群、片区、聚落、公共组织结构等。
2. **多尺度交付信号**
   - 如总体规划、建筑语言、公共空间、室内系统并存。
3. **多单元组织信号**
   - 如房量结构、业态配比、公共节点、运营单元等。

注意：

- 第 1 层只能识别“聚落级问题存在”，不得提前替代第 2-3 层做正式类型定性与框架选择；
- 不得把“集群”直接简化成“多个客房”或“普通酒店项目”；
- 不得把总体规划尺度要求吞没进单体室内任务。

#### 5.7.11 第 1 层需识别“文化—产业—经济”复合变量输入

经 `sf/q.txt` 第 3 条测试确认，第 1 层必须支持一类非纯空间参数型输入：

- **文化变量**
- **产业变量**
- **经济逻辑变量**

典型表现包括：

- 在地农耕文化
- 乡村产业结构
- 乡村经济逻辑
- 文旅导入与商业成功并行要求

这类内容当前至少应进入：

- `reference_pixels`
- `goal_pixels`
- `tension_signal_pixels`
- `gap_pixels`

说明：

- 这类变量不能只被当作“项目背景介绍”；
- 它们是后续框架调用层与任务框架层的重要前置输入；
- 第 1 层必须先把它们显式拆出来，防止后续只剩审美叙事而失去真实项目逻辑。

#### 5.7.12 第 1 层需识别“高阶目标词”的语义缺口

经 `sf/q.txt` 第 3 条测试确认，第 1 层除识别价值判断词外，还必须识别一类更宏观的语义缺口：

> **高阶目标词语义缺口**

即：brief 中出现“示范意义”“项目模型”“文化深度”“商业成功”等高阶目标词，但其真实评价标准、落地方式与尺度归属尚未定义清楚。

典型例子包括：

- “文化示范意义”
- “项目模型”
- “文化深度”
- “商业成功”

如果系统不能识别这类缺口，就会过早把宏观口号当成稳定任务定义，导致后续层级出现：

- 目标尺度混乱
- 评价标准缺失
- 概念过大而执行抓手不足

#### 5.7.13 第 1 层需支持“品牌气质转译型输入”

经 `sf/q.txt` 第 4 条测试确认，第 1 层还必须稳定识别一类介于品牌、场地与度假场景之间的复合输入：

- **品牌气质转译型输入**
- **品牌语言 × 场地语境双约束输入**
- **轻度度假型生活方式输入**

典型表现包括：

- brief 以某品牌的气质、态度、价值倾向作为核心起点；
- 同时要求回应真实场地资源、自然景观与使用场景；
- 项目既要保留品牌辨识度，又不能压扁场地自身语境；
- 用户要求输出色彩、家具、尺度、社交组织等系统性内容，但第 1 层仍只能停留在拆解。

对这类输入，第 1 层至少要能拆出以下内容：

1. **品牌气质信号**
   - 如年轻、克制、功能主义、轻快、日常性、非炫耀性等。
2. **场地语境信号**
   - 如山地、自然景观、度假场景、在地环境约束等。
3. **双约束目标信号**
   - 如品牌辨识度 vs 场地适配性、功能主义 vs 度假松弛感、克制表达 vs 体验丰富度。

注意：

- 第 1 层只能将上述内容记录为 `reference_pixels`、`goal_pixels`、`tension_signal_pixels` 与 `gap_pixels` 的组成部分；
- 不得直接把品牌气质翻译成色彩答案、家具清单或审美结论；
- 不得把“年轻”“克制”“功能主义”“轻度度假”当成已定义清楚的稳定事实。

#### 5.7.14 第 1 层需识别“气质词 / 场景词”的语义缺口

经 `sf/q.txt` 第 4 条测试确认，第 1 层除识别价值语义缺口与高阶目标语义缺口外，还必须识别：

> **气质词 / 场景词语义缺口**

即：brief 中使用了看似明确的品牌气质词、生活方式词或度假场景词，但其真实含义、边界与转译方式尚未被定义清楚。

典型例子包括：

- “年轻”
- “克制”
- “功能主义”

   - 如留守儿童、老人群体、志愿者等。
3. **持续运营信号**
   - 如志愿者活动组织机制、普惠覆盖 vs 运维持续性、网络覆盖与活动责任待明确等。

注意：

- 第 1 层不得把该类输入降格为单一儿童阅览室或普通社区活动室；
- 不得把志愿机制默认为稳定成立事实；
- 必须保留“网络系统”而不是只摘取某一个节点空间程序。

#### 5.7.42 第 1 层需支持“非遗产业聚落型输入”

经 `sf/q.txt` 第 161 条测试确认，第 1 层还必须稳定识别一类由传统手工艺、游客体验、销售转化与产业孵化共同构成的复合输入：

- **非遗产业聚落型输入**
- **传统工艺聚落型输入**
- **展示—体验—转化—孵化复合输入**

典型表现包括：

- brief 以陶瓷、木雕、刺绣等传统手工艺为参考锚点；
- 输入同时要求工坊展示界面、游客体验路径、销售转化空间与产业孵化机制；
- 问题核心是非遗生产、展示、体验、转化与孵化的链条协同，而不是旅游商业包装。

对这类输入，第 1 层至少要能拆出以下内容：

1. **非遗参考信号**
   - 如非遗、陶瓷、木雕、刺绣、传统手工艺等。
2. **链条程序信号**
   - 如工坊展示、游客体验、销售转化、产业孵化等。
3. **聚落边界信号**
   - 如文化展示体验 vs 产业孵化效率、生产展示销售孵化四类空间配比待明确等。

注意：

- 第 1 层不得把该类输入退化为文创店、旅游纪念品街区或普通展陈题；
- 不得把“非遗”直接翻译成审美符号拼贴；
- 必须保留聚落尺度与产业链条，而不是只截取游客体验面。

#### 5.7.43 第 1 层需支持“保障住房品质示范型输入”

经 `sf/q.txt` 第 167 条测试确认，第 1 层还必须稳定识别一类由保障性供给、公平品质提升与长期维护成本控制共同构成的复合输入：

- **保障住房品质示范型输入**
- **高品质保障房输入**
- **公平供给 × 品质升级型输入**

典型表现包括：

- brief 同时提出高品质、保障房、示范项目、打破低端形象等目标；
- 输入要求标准化模块优化、公共共享空间升级与长期维护成本控制并行；
- 问题核心是品质感、公平性与维护成本之间的平衡，而不是单纯住宅升级题。

对这类输入，第 1 层至少要能拆出以下内容：

1. **制度与品质目标信号**
   - 如保障房、示范项目、高品质、打破低端形象等。
2. **标准化程序信号**
   - 如标准化模块优化、公共共享空间升级、长期维护成本控制等。
3. **评价与边界信号**
   - 如品质提升 vs 成本控制、品质感表达与保障性公平边界待明确等。

注意：

- 第 1 层不得把该类输入降格为普通商品房精装优化题；
- 不得把“高品质”直接写成已定义清楚的评价事实；
- 必须同时保留制度属性、品质目标与长期运维边界。

### 5.8 第 1 层新增失败样式（基于近期测试）

13. 把“文化示范意义”“项目模型”“文化深度”“商业成功”直接当成已定义事实，而不识别其为高阶目标语义缺口。

## 15. 专题补充：需求分析 × 任务节点关系规范（主架构增强版）

> 生效日期：2026-03-08  
> 用途：作为“一级大节点关系哲学与执行口径”的专题补充章。  
> 约束：本章以当前主架构为唯一主轴，参考意见 B/C/D/E 仅用于局部补强、盲点交叉验证与风险提醒；若与主架构冲突，以本文件既有冻结口径与 `CLAUDE.md` 约束为准。

### 15.1 节点关系总定义（当前冻结口径）

本系统在“需求分析”与“任务落实”两大一级节点之间，正式采用以下关系定义：

1. **需求分析 = 任务书框架生成器**
   - 产出的是：目标、边界、维度、倾向、建议、可能性与留白；
   - 不产出：唯一执行路线、唯一风格答案、唯一实现细案。
2. **任务落实 = 框架内创造性展开器**
   - 负责：延展、细化、深入、补充、扩展、路径化；
   - 禁止：机械抄写上游文本、把任务书降格为章节扩写模板。
3. 两者关系不是“上游命令 + 下游誊写”，而是：
   - **框架约束 + 创造展开**；
   - **定势 + 生长**；
   - **引子 + 生成**。

### 15.2 交接协议（四层合同）

为防止“前序越权写死”与“后序无边界乱飞”两种极端，当前确定在节点交接处采用四层协议：

1. **硬约束层（必须继承）**
   - 内容：核心目标、红线边界、显性硬约束、项目类型主语境。
   - 规则：任务节点不得突破。
2. **判断重力层（优先参考）**
   - 内容：倾向判断、重心建议、关键维度、风险提醒。
   - 规则：任务节点可重排优先级，但需保持语义同向。
3. **工作假设层（允许扩展）**
   - 内容：待验证动机、潜在驱动、替代切入点、跨学科补位线索。
   - 规则：任务节点应主动验证、补充或反证。
4. **生成空位层（鼓励创新）**
   - 内容：刻意留白、未封口矛盾、待二次判断区域。
   - 规则：任务节点可在边界内主权生长，不要求唯一答案。

### 15.3 参考建议 B/C/D/E 交叉验证结论

#### 15.3.1 采纳矩阵

1. **建议 B（高自主治理）**：高采纳
   - 采纳：框架约束 + 创造展开；允许优先级重排、子任务新增、方法替代、路径扩展。
   - 部分采纳：不强制偏离声明可接受，但需最小保护（见 15.4）。
2. **建议 C（权力关系重构）**：高采纳
   - 采纳：需求分析不是答案半成品；任务节点是第二判断层。
   - 处理：将哲学表述转译为可执行协议与验收规则，避免停留口号。
3. **建议 D（重力场与留白哲学）**：高采纳
   - 采纳：需求分析交付“势能/重力场”，任务节点交付“展开路径”。
   - 处理：采用“四层交接协议”作为系统级落地锚点。
4. **建议 E（任务书 vs 施工图）**：中高采纳
   - 采纳：定方向不定答案、划边界不填内容。
   - 处理：补充“何时纠偏/何时放行”的工程判据。

#### 15.3.2 冲突排查（必须显式处理）

1. **高自主 vs 可追溯性**
   - 冲突：高自主若完全无记录，难区分“高质量偏离”与“跑题偏离”。
   - 处置：首发阶段不强制记录，但保留“可选扩展说明区”；若指标恶化再进入二阶段轻量记录。
2. **不强制回流需求分析 vs 方向漂移风险**
   - 冲突：全部本地裁决可能累积偏移。
   - 处置：不改“高自主”前提，纠偏优先放在审批节点与追问节点，不强制回流 `requirements_analyst`。
3. **多路径并存 vs 审批稳定性**
   - 冲突：同题多解会造成审批口径波动。
   - 处置：审批采用“最小一致性判据”（见 15.4.2），不以路径唯一性作为通过条件。

### 15.4 高自主版本的执行思路（当前推荐）

#### 15.4.1 治理总口径

1. 一阶段采用“**中高自主 + 结果质量优先**”策略。
2. 二阶段仅在指标恶化时启用“**轻量偏离记录**”。
3. 不引入强制回流与重审计，避免把任务节点重新压回机械执行器。

#### 15.4.2 最小保护三件套（不违背高自主）

1. **审批节点：框架一致性检查摘要（必须）**
   - 只核三项：
     - 是否命中任务书核心目标；
     - 是否突破硬边界；
     - 是否遗漏关键维度。
2. **任务输出：扩展说明区（可选）**
   - 鼓励说明新增任务/替代方法的价值；
   - 不设硬性结构化留痕门槛。
3. **追问节点：偏移优先纠偏（必须）**
   - 对因高自主产生的偏题或尺度失衡，优先在追问阶段收敛；
   - 不默认回退需求分析节点。

#### 15.4.3 任务节点权限包（中高自主）

任务节点在框架内可执行以下动作：

1. 优先级重排；
2. 子任务新增；
3. 方法替代；
4. 交付路径扩展；
5. 限度反驳前序弱假设（不得突破硬边界）。

### 15.5 测试场景与通过标准（专题版）

沿用并固化以下四类关系测试：

1. **场景 A：大幅扩展但边界内**
   - 期望：通过。
2. **场景 B：偏离倾向但结果更优**
   - 期望：审批可放行，不强制回流。
3. **场景 C：扩展导致偏题**
   - 期望：在审批或追问节点纠偏。
4. **场景 D：同题多路径并存**
   - 期望：允许非唯一执行路线。

### 15.6 二阶段触发条件（仅当需要时启用）

若出现以下任一趋势，应启动“轻量偏离记录”而非重管控回退：

1. 偏题率持续上升；
2. 审批驳回率持续上升；
3. 返工率持续上升；
4. 多轮追问仍无法收敛关键边界。

触发后仅新增最小字段：`deviation_note`（短文本）与 `deviation_reason_tag`（枚举标签），不引入重型过程审计链。

### 15.7 与既有冻结约束的一致性声明

1. 本章不恢复 `progressive_step1_core_task`，不修改其废弃结论。
2. 本章不改变 `unified_input_validator_secondary -> progressive_step3_gap_filling` 锁定关系。
3. 本章不重排主链，仅补充“需求分析 -> 任务落实”之间的治理口径与交接协议。
4. 若未来执行层改造与本章冲突，优先以本文件与 `CLAUDE.md` 冻结记录做排查，再决定是否升级协议。

## 16. 执行模板包：审批一致性摘要与任务扩展说明（可直接接线）

> 生效日期：2026-03-08  
> 用途：给评审节点与任务节点提供可直接落地的“轻治理模板”。  
> 原则：不引入强制重审计；保持高自主；仅做最小保护与一致性收敛。

### 16.1 审批节点模板 A：框架一致性检查摘要（必须）

#### 16.1.1 模板目标

在不要求任务节点提交重型过程留痕的前提下，审批节点只回答一个问题：

> 当前任务展开是否仍在任务书框架内，并且结果质量可放行？

#### 16.1.2 模板结构（建议 JSON）

`framework_consistency_summary`：

1. `core_goal_alignment`
   - 含义：是否命中任务书核心目标。
   - 取值：`pass` / `risk` / `fail`。
2. `hard_boundary_integrity`
   - 含义：是否突破硬边界（预算上限、合规红线、实施硬约束、制度边界）。
   - 取值：`pass` / `risk` / `fail`。
3. `key_dimension_coverage`
   - 含义：关键维度是否被覆盖（功能、情绪、人本、运营、风险等按题型适配）。
   - 取值：`pass` / `risk` / `fail`。
4. `overall_decision`
   - 含义：审批结论。
   - 取值：`approve` / `approve_with_notes` / `needs_correction`。
5. `decision_rationale`
   - 含义：简短理由（1-3 条）。
6. `correction_focus`
   - 含义：若需纠偏，指出优先纠偏点（可为空）。
7. `path_diversity_note`
   - 含义：若同题多路径并存，说明为何允许并存（可为空）。

#### 16.1.3 最小判定规则（审批端）

1. 任一项为 `fail` → `overall_decision = needs_correction`。
2. 全部 `pass` → `overall_decision = approve`。
3. 存在 `risk` 但无 `fail` → `overall_decision = approve_with_notes`。
4. 对“偏离倾向但结果更优”的情况，允许 `approve_with_notes` 或 `approve`，不强制回流需求分析。

### 16.2 任务节点模板 B：扩展说明区（可选）

#### 16.2.1 模板目标

该说明区用于鼓励高质量扩展被理解与复用，而非作为放行前置门槛。

#### 16.2.2 模板结构（建议 JSON）

`optional_extension_note`：

1. `extension_trigger`
   - 本次扩展触发原因（如：目标冲突、维度缺口、实施约束、新增高价值机会）。
2. `what_was_extended`
   - 扩展内容（新增子任务 / 路径扩展 / 方法替代 / 优先级重排）。
3. `expected_value`
   - 预期价值（质量提升、风险下降、可实施性提升、长期运营收益）。
4. `boundary_check`
   - 扩展后对边界影响（是否仍满足核心目标与硬约束）。
5. `tradeoff_note`
   - 关键取舍（如短期效率 vs 长期质量）。

#### 16.2.3 使用规则

1. 该区块默认可空，不提交不影响审批流程进入。
2. 若提交，应优先用于评审沟通与复盘，不作为硬性打分项。
3. 若后续触发二阶段治理，可将其升级为“轻量偏离记录”的来源之一。

### 16.3 审批放行标签（推荐）

为避免“放行理由失忆”，审批节点建议输出 1-2 个标签：

1. `within_framework_high_quality`
   - 框架内高质量延展。
2. `beneficial_deviation`
   - 偏离倾向但结果更优。
3. `multi_path_valid`
   - 同题多路径均可成立。
4. `boundary_risk_watch`
   - 当前可放行但需关注边界风险。
5. `needs_followup_questioning`
   - 需在追问阶段补齐关键缺口。

### 16.4 追问节点纠偏模板（必须支持）

当审批输出 `needs_correction` 或 `boundary_risk_watch` 时，追问节点优先使用以下结构：

`followup_correction_payload`：

1. `deviation_type`
   - `goal_drift` / `boundary_break` / `dimension_missing` / `evaluation_gap`。
2. `must_fix_items`
   - 必修项列表（1-3 条）。
3. `optional_refine_items`
   - 可优化项列表（0-3 条）。
4. `acceptance_recheck_rule`
   - 复核口径（修正后满足哪些条件可放行）。

说明：该模板用于“本地纠偏收敛”，不默认触发回流 `requirements_analyst`。

### 16.5 与现有对象链的映射建议

本章模板建议接到以下对象消费位：

1. `RequirementsInsight`（定稿输出）
   - 可挂载：`framework_consistency_summary`（审批结果）
   - 可挂载：`optional_extension_note`（可选）
2. `TaskApprovalMapping`
   - 可挂载：`overall_decision`、`approval_tags`、`correction_focus`
3. `questionnaire_summary.py` / `project_director.py`
   - 保持“优先读新对象、回退读旧字段”兼容策略。

### 16.6 场景化示例（简化）

1. **场景 A（大幅扩展但边界内）**
   - 审批摘要：`pass/pass/pass -> approve`。
   - 标签：`within_framework_high_quality`。
2. **场景 B（偏离倾向但结果更优）**
   - 审批摘要：`pass/risk/pass -> approve_with_notes`。
   - 标签：`beneficial_deviation`。
3. **场景 C（扩展导致偏题）**
   - 审批摘要：`fail/risk/risk -> needs_correction`。
   - 追问：`deviation_type = goal_drift`，给出 `must_fix_items`。
4. **场景 D（同题多路径成立）**
   - 审批摘要：`pass/pass/pass -> approve`。
   - 标签：`multi_path_valid`。

### 16.7 升级策略（保持轻治理）

若后续指标恶化（偏题率/驳回率/返工率持续上升），仅按以下顺序小步增强：

1. 先启用 `deviation_note` 短文本；
2. 再启用 `deviation_reason_tag` 枚举；
3. 最后才考虑扩展审计字段；
4. 不得直接回退到强制重流程与强制回流模式。

## 17. 基于新架构的需求分析师优化反思（执行版）

> 生效日期：2026-03-08  
> 目标：在“需求分析定势、任务节点生长”的新架构下，反思 `requirements_analyst` 模块仍存在的结构性短板，并给出可执行优化顺序。

### 17.1 当前主要短板（客观评估）

结合当前实现与第 15/16 章口径，需求分析师仍有以下 8 类优化空间：

1. **框架输出过强文本化，任务可执行语义不足**
   - 现象：深度分析丰富，但“可直接被任务节点消费的任务语义槽位”仍不稳定。
   - 影响：任务节点容易回到“读长文再二次猜测”的模式。

2. **第 1 层到第 6 层的“信息压缩率”缺少显式控制**
   - 现象：`ProblemPixels` 非常细，但转入 `TaskFramework` 时存在信息损耗风险。
   - 影响：高价值信号（如语义缺口、关键张力）可能在中段丢失。

3. **M22/M23/M24 对任务生成的驱动力仍偏弱**
   - 现象：立场、叩问、授权在分析层表达充分，但向任务层的“可执行约束/授权”映射还不够硬。
   - 影响：任务延展的主观能动性与边界控制连接不够紧。

4. **“留白”有理念，但缺少机器可判定边界**
   - 现象：已强调留白，但没有统一标识哪些是“必须继承”、哪些是“鼓励扩展”。
   - 影响：不同任务节点对同一输入可能出现口径漂移。

5. **跨题型稳定性高，但跨尺度稳定性仍需增强**
   - 现象：题型覆盖广（项目/人物/研究/治理等），但从房间→单体→聚落→网络系统的尺度跳变仍有退化风险。
   - 影响：高复杂项目可能在中后段被“单体化”。

6. **风险识别已有扩展，但风险分级未统一**
   - 现象：已覆盖误读/实施/运维/合规风险，但缺少统一分级与传递优先级。
   - 影响：下游难判断“先修什么”。

7. **审阅层可见性增强，但“可纠偏动作”仍可更具体**
   - 现象：review 能看到问题，但对“下一步怎么改”指令化不足。
   - 影响：审批与追问之间可能出现重复沟通成本。

8. **测试样本数量高，但“失败归因模板”尚未标准化**
   - 现象：失败样式非常丰富。
   - 影响：难以快速聚合“同类失败 -> 对应修复策略”。

### 17.2 优化目标（对齐新架构）

需求分析师优化后应达到以下状态：

1. 输出的是“可执行任务框架底稿”，不是“高质量分析散文”。
2. 明确区分“继承区 / 参考区 / 扩展区 / 挑战区”（四层交接协议可机器消费）。
3. 让任务节点具备中高自主扩展权，同时由审批与追问做轻量收敛。
4. 保持主链不回退、不重排，全部改造均以前向增强方式进行。

### 17.3 优先级路线（P0 / P1 / P2）

#### P0（立即优化，低风险高收益）

1. **输出合同强化**
   - 在 `TaskFramework` 增加四层交接区块：
     - `must_inherit` / `prefer_follow` / `allow_extend` / `encourage_challenge`。
2. **M22/M23/M24 映射强化**
   - 将 `stance_spectrum`、`expert_handoff_questions`、`disruption_authorization` 显式映射到任务框架约束与授权位。
3. **风险分级统一**
   - 引入 `risk_level = low/medium/high` 与 `risk_priority = p0/p1/p2` 的最小双轴。
4. **review 纠偏动作模板化**
   - 对接第 16 章 `followup_correction_payload`，让每条风险至少对应 1 条可执行纠偏动作。

#### P1（阶段优化，中风险中高收益）

1. **信息压缩守恒检查**
   - 在 `ProblemPixels -> TaskFramework` 之间新增“关键信号守恒检查”，避免核心张力与语义缺口在转译时丢失。
2. **跨尺度稳定性增强**
   - 对“聚落级 / 网络级 / 平灾双模式”题型增加尺度守护规则，避免降格为单体摘要。
3. **失败归因标准化**
   - 建立失败归因标签：`boundary_miss`、`dimension_drop`、`semantic_freeze`、`scale_collapse`、`over_stylization`。
4. **审批摘要自动生成联动**
   - 在 `RequirementsInsight` 输出时自动附带 `framework_consistency_summary` 草稿供审批节点复核。

#### P2（治理增强，高收益按需启用）

1. **轻量偏离记录（二阶段）**
   - 指标恶化后启用 `deviation_note` + `deviation_reason_tag`。
2. **高价值偏离学习库**
   - 沉淀“偏离倾向但结果更优”的样本，用于优化任务节点扩展策略。
3. **任务路径多样性评估**
   - 对同题多解输出进行质量对比，避免“唯一答案幻觉”回潮。

### 17.4 文件级优化建议（与现有代码位对齐）

1. `agents/requirements_analyst_agent.py`
   - 强化四层交接区块输出；
   - 增加 M22/M23/M24 到任务语义位的映射函数。
2. `interaction/nodes/requirements_restructuring.py`
   - 增加“关键信号守恒检查”；
   - 明确从分析语义到任务槽位的转译规则。
3. `interaction/nodes/requirements_analyst_review.py`
   - 增加失败归因标签输出与纠偏动作建议。
4. `interaction/nodes/questionnaire_summary.py`
   - 输出 `RequirementsInsight` 时联动第 16 章审批摘要草稿。
5. `agents/project_director.py`
   - 优先消费四层交接区块与审批摘要，减少对散字段依赖。

### 17.5 验收标准（新增）

完成本章优化后，至少满足：

1. 任一高复杂输入都能稳定输出四层交接区块。
2. M22/M23/M24 至少 80% 能在 `TaskFramework` 找到对应消费位。
3. 审阅输出中每个 `high` 风险都对应至少 1 个 `must_fix_items`。
4. 聚落级/网络级输入不再被高频降格为单体任务摘要。
5. 场景 B（偏离但更优）可被审批稳定放行，不强制回流需求分析。

### 17.6 风险提醒与边界

1. 本章优化不得回退已冻结主链，不得恢复废弃 step1 节点。
2. 本章优化不得将高自主策略重新硬化为强审计主导模式。
3. 若出现质量波动，先启用第 16 章轻治理增强，再考虑结构改造。

## 18. 方法论澄清：七层结构是不是核心？还有哪些需要优化

> 生效日期：2026-03-08  
> 用途：回答“需求分析师核心方法论到底是什么”的口径问题，避免团队把单一结构误当全部方法论。

### 18.1 结论（冻结口径）

1. **七层结构是核心方法论之一，但不是唯一方法论。**
2. 七层结构解决的是：
   - 从输入到任务框架的**认知装配顺序**；
   - 需求分析节点内部“如何看清问题并形成可交接对象”的主干问题。
3. 需求分析师的完整方法论，应理解为“**结构方法 + 治理方法 + 质量方法 + 演进方法**”的组合，而非单一七层框架。

### 18.2 需求分析师当前方法论全景（建议作为团队共识）

除七层结构外，至少还包括以下 6 组方法论：

1. **四层交接协议方法（第 15 章）**
   - `must_inherit / prefer_follow / allow_extend / encourage_challenge`；
   - 解决“任务节点如何在框架内创造”问题。

2. **高自主轻治理方法（第 15/16 章）**
   - 高自主优先，审批与追问做最小收敛；
   - 解决“防机械执行”与“防失控”之间平衡问题。

3. **27 方法机制线 + 一致性门禁方法**
   - 通过 execution trace 与门禁规则保障分析完整性与最低质量线；
   - 解决“分析是否可信、是否缺关键方法”问题。

4. **问题测试先行方法（第 3.3）**
   - `问题测试 -> 反思补缺 -> 文档修正 -> 再编码`；
   - 解决“架构未稳先编码导致返工”问题。

5. **桥接对象渐进冻结方法（第 4/6/9）**
   - 先桥接收口，再稳定消费，最后协议冻结；
   - 解决“要么散字段、要么过早重工程化”问题。

6. **优先读新对象、回退读旧字段兼容方法**
   - 保证前向增强，不破坏主链与既有回归；
   - 解决“演进与稳定性冲突”问题。

### 18.3 当前是否需要更新优化？结论：需要，且是“组合式优化”

不建议继续争论“只靠七层是否足够”，应直接进入组合式优化：

1. **七层结构继续保留为主干，不替换。**
2. **补齐方法论外环：**
   - 交接协议显式化（四层）；
   - 审批摘要与追问纠偏模板化；
   - 风险分级与失败归因标准化；
   - 指标触发的二阶段轻记录。
3. **优化目标不是增加概念数量，而是提升“可执行性与可收敛性”。**

### 18.4 建议更新项（按优先级）

#### P0（必须）

1. 在 `TaskFramework` 显式输出四层交接区块。
2. 将 M22/M23/M24 映射为任务层可执行语义位。
3. 在 `RequirementsInsight` 挂载 `framework_consistency_summary`。

#### P1（应做）

1. 增加 `ProblemPixels -> TaskFramework` 关键信号守恒检查。
2. 标准化失败归因标签并接入 review。
3. 统一风险分级（`risk_level` + `risk_priority`）。

#### P2（按指标触发）

1. 启用 `deviation_note` 与 `deviation_reason_tag`。
2. 建立“高质量偏离样本库”。
3. 做同题多路径质量对比评估。

### 18.5 口径误区（需避免）

1. 把“七层结构”误解为“唯一方法论”。
2. 把“高自主”误解为“无需边界检查”。
3. 把“过程简化”误解为“不需要纠偏机制”。
4. 把“对象增多”误解为“方法更先进”。

### 18.6 一句话原则（团队传播版）

> 需求分析师的核心不是“七层本身”，而是“七层主干 + 交接协议 + 轻治理收敛 + 渐进冻结”的组合方法论。
