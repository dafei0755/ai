## 30 秒版本：这个系统是什么

**Intelligent Project Analyzer** 是一个多 Agent 协作的设计项目需求分析系统。

**用户场景**：室内设计师/产品设计师/品牌设计师在接到项目前，需要系统性地理解客户需求。系统接受用户输入（文字需求、文件、问卷回答），经由 6 个专业 AI 专家并行分析，输出结构化的需求报告（含雷达图维度评分）。

**核心技术价值**：用 LangGraph 状态机正确编排了多 Agent 并行流程，并在工作流中嵌入了人工暂停介入机制（追问问卷），使系统能在拿到不充分信息时主动向用户补充提问，而不是凭猜测生成低质量报告。

**技术架构一句话**：LangGraph 驱动 6 个并行专家 Agent → FastAPI + WebSocket 实时推送 → Next.js 14 前端展示。

## 做对了什么

### 1. LangGraph 框架选型正确
`ProjectAnalysisState`（TypedDict + 自定义 reducer）、Send API 并行扇出、Interrupt 人工暂停恢复、SqliteSaver checkpoint——这些都是 LangGraph 框架能力的正确应用。状态机编排 + checkpoint 机制使工作流可恢复，是 LLM 多步骤工作流的正确技术方向。

### 2. 质量控制前置化（ADR-003）
将 `role_selection_quality_review` 合并到 `role_task_unified_review`，在任务分派阶段就检测质量问题，使专家不会白做。这个决策正确，且应该更早做。

### 3. asyncio.gather 真并行（ADR-002）
绕过 LangGraph Send API 并不真正并行的问题，在节点内部直接用 `asyncio.gather` 实现真并行。实测 Batch1 从 10s 降至 3.3s（67% 加速）。关键教训：框架的"并行" API 不代表底层真并行，要实测。

### 4. 生产级可观测性
Grafana + Loki + Promtail + Prometheus + Loguru 结构化日志——在功能开发阶段就投入了生产级监控，这是正确的优先级判断。

### 5. 双道安全校验
输入安全校验（`security_check_node`）+ 报告输出安全校验，安全层独立成模块。设计思路合理，实现可靠。

### 6. v8.0.0 版本治理基线（ADR-005）
在 v8.0.0 建立了 SSOT 机制（VERSION 文件 + SemVer），这是系统治理的正确起点。虽然来得太晚，但建立了就有了基础。

## 做错了什么

### 1. 废弃节点清理不彻底（最直接的代价：P0 崩溃风险）
v2.2 废弃了 `analysis_review` 节点，但路由函数中的返回值和孤立函数没有同步清理，导致死代码在 `requires_client_review == True` 时触发 LangGraph 崩溃。从废弃决策到问题发现，中间历经了数十个版本。

**根本原因**：没有"废弃检查清单"，废弃动作是临时性的、不完整的。

### 2. 版本标记滥用（500+ 处 v7.xxx 注释）
整个代码库内遗留 500+ 处 `# v7.xxx 修改...` 风格注释，形成认知噪音。新维护者无法判断哪些注释仍然有效，哪些只是历史遗迹。

**根本原因**：没有建立版本注释规范（什么时候加、什么时候删）。

### 3. server.py 单文件无限扩张
`api/server.py` 增长到 8534 行，启动/WebSocket/会话/上传/认证全混杂。虽然 `routes/` 子目录已开始拆分，但执行不彻底——每次新功能都直接加到 `server.py`，拆分动作被推迟。

**根本原因**：缺少明确的"单文件行数警戒线"（应该在 500 行时就开始拆分）。

### 4. 状态字段膨胀（4 个 DEPRECATED 字段，同一概念多字段表达）
`state.py` 还有 4 个 DEPRECATED 字段没有清理；同一概念（问卷答案）用 `calibration_answers`、`questionnaire_responses`、`gap_filling_answers` 三个不同字段表达，造成混乱。

**根本原因**：没有状态字段变更的管理规范（增减字段需要明确的生命周期）。

### 5. 技术债务 > 70%
根目录 45 个临时脚本、`tests/` 80+ 文件平铺、ruff 规则未激活、测试覆盖率仅 11%——这些不是一时疏忽，而是长期在功能迭代的压力下推迟了工程治理工作。

**根本原因**：没有"治理债务不过版本"的原则，导致债务在每次迭代中积累而非偿还。

## v8.1.0 应该优先做什么

按照 [ACTION_PLAN.md](ACTION_PLAN.md) 阶段 0+1：

1. **阶段 0（< 2h）**：QW-1 删除死代码路由 → QW-2 添加并发 Semaphore → QW-3 删除孤立函数
2. **阶段 1（约 3 天）**：ST-1 清理 DEPRECATED 字段 → ST-2 清除版本开关 → ST-3 节点 Fallback 守卫 → ST-4 激活 lint → ST-5 脚本归类

**不应该**在 v8.1.0 做的事：server.py 完整拆分（MT-1）、main_workflow.py 拆分（LT-1）——这些是 P2/P3 工作，应在 v8.2.0+ 计划性推进，不要在偿还 P0/P1 债务的同时引入大范围重构风险。

## 给未来的自己：4 条核心原则

> 用于每次迭代开始时自我检查

**原则 1：废弃即清理，不留"稍后处理"**
废弃任何节点/字段/函数时，当场执行[节点废弃 5 步清单](DEV_WORKFLOW.md#节点废弃-5-步清单)。"先废弃，稍后清理"是 P0 死代码路由的制造原因。

**原则 2：单文件超 500 行立刻拆分**
不要等到 8534 行。在第 500 行时，就开始思考拆分边界。

**原则 3：P0 问题不进入下一个 MINOR**
P0（已知崩溃风险）必须在当前 MINOR 版本内修复。P1（影响可维护性）可以在下一个 MINOR 内处理，但不能跳过一个版本。

**原则 4：治理债务与功能债务同等优先**
每次迭代的工作配比建议：新功能 60% + 技术债务偿还 30% + 文档/测试 10%。不要等到"v8 大版本"才集中清理——那时的代价是现在的 3-5 倍。
