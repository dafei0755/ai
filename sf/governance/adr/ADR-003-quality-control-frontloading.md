# ADR-003：质量控制前置化

**状态**：已完成，生效中
**决策日期**：v7.280
**记录日期**：2026-02-28（v8.0.0 治理审计时补录）

---

## 背景

系统早期的质量控制架构是**后置式**的：

```
角色选择 → 任务分配 → 专家执行（消耗大量 LLM Token）→ 质量审核（由 analysis_review 节点）
```

`role_selection_quality_review` 节点独立于任务分配之后执行，导致以下问题：

1. **专家计算资源浪费**：专家已经基于"质量不合格的任务定义"完成了全量 LLM 调用才被审核发现问题
2. **延迟重做代价高**：发现问题后需要重新分配任务并再次执行专家，延迟倍增
3. **用户体验差**：整个工作流失败后，用户感知的延迟极长

---

## 决策

将 `role_selection_quality_review` 合并入 `role_task_unified_review` 节点，使质量审核**在任务分配阶段**完成，而非在专家执行之后。

新的流程：

```
角色选择 → role_task_unified_review（同时完成任务分配 + 质量审核）
         → 通过 → 专家执行
         → 不通过 → 重新分配任务（廉价）
```

---

## 实现细节

- `role_task_unified_review` 节点：在一次 LLM 调用中同时完成任务质量评估和任务分配决策
- 质量不合格时：路由回任务分配节点，代价仅是一次轻量分配调用（而非 10-15 个专家全量重跑）
- `analysis_review` 节点：因本决策而失去存在意义，在 v2.2 被废弃（见 ADR-001）

---

## 后果

**积极后果**：
- 质量问题在早期阶段被发现，避免了专家资源浪费
- 系统整体效率提升（失败代价从"专家全量重跑"降为"任务分配层重试"）
- 流程结构更清晰（质量校验与任务分派合并为一个决策节点）

**遗留的消极后果**：
- `analysis_review` 节点废弃时（ADR-001），路由清理不彻底，遗留了 P0 问题

---

## 教训

**教训 1**：质量控制越前置，浪费的计算资源越少。这个原则不仅适用于本系统，也适用于所有多步骤 LLM 流水线的设计。

**教训 2**：应该更早做这个决策。v7.0 版本设计时就应该考虑质量前置，而不是在 v7.280 才意识到后置质量控制的代价。

**推论**：设计新的多步骤 LLM 流程时，默认应将质量/验证环节放在"最廉价的转折点之前"，而不是"最昂贵的执行步骤之后"。
