# ADR-005：版本治理 SSOT + SemVer 规范

**状态**：已决策，v8.0.0 起生效
**决策日期**：v8.0.0（2026-02-27）
**记录日期**：2026-02-28

---

## 背景

### 历史版本号混乱

从 v7.0 演进到 v8.0.0 的过程中，版本号管理完全缺乏规范：

- **版本号跨度**：v7.0 ~ v7.999.9（中间经历了数百个小版本，数字含义不一致）
- **无 SSOT**：版本号分散在 `setup.py`（如果存在）、代码注释、CHANGELOG、README 中，各处可能不一致
- **标记滥用**：代码库中有 500+ 处 `# v7.xxx 修改...` 风格注释，版本与功能/修复的对应关系不清晰
- **无 Tag**：git 仓库中缺乏对应的版本 tag，"回滚到某个版本"的操作成本极高

### 问题后果

- 维护者无法快速判断"当前运行的版本是什么"
- CHANGELOG 中的版本号与代码实际状态可能不一致
- 新功能开发时无法确定应该将版本号从哪里读取

---

## 决策

v8.0.0 起建立版本治理单一事实来源（SSOT）机制：

### 决策 1：SSOT 为根目录 `VERSION` 文件

```
# VERSION 文件内容示例
8.0.0
```

所有读取版本号的地方（API 端点、启动日志、README 徽章、打包配置）均从此文件读取，不硬编码。

### 决策 2：采用 SemVer 语义化版本

- `PATCH`：Bug 修复、文档更新，不改 API
- `MINOR`：新功能、重构，向后兼容；**每个 MINOR 版本打一个 git tag**
- `MAJOR`：破坏性变更（State 字段删除、API 签名变更）

### 决策 3：CHANGELOG 结构规范

采用 [Keep a Changelog](https://keepachangelog.com/zh-CN/1.0.0/) 格式：

```markdown
## [Unreleased]
### Added
### Changed
### Fixed
### Deprecated
### Removed

## [8.0.0] - 2026-02-27
### Changed
- ...
```

**发布时**将 `[Unreleased]` 改为 `[版本号] - 日期`，新建空的 `[Unreleased]` 节。

### 决策 4：代码中不再出现 `v7.xxx` 风格标记

v8 时代起，代码注释中：
- 架构变更 → 引用 ADR 编号（如 `# see ADR-001`）
- 功能变更 → 一行说明（如 `# 引入混合维度策略，v8.1.0`）
- Bug 修复 → 只在 CHANGELOG，代码中不留标记

CI 门禁（待实现）：检测新 commit 中是否引入了 `v7.` 风格标记。

---

## 后果

**积极后果**：
- 版本信息单一来源，不一致问题不再发生
- git tag 使回滚操作具有明确的锚点（`git checkout v8.0.0 -- .`）
- 代码注释噪音将逐渐减少（存量标记逐步清理，新增被门禁阻止）

**遗留问题**：
- 存量 500+ 处 `v7.xxx` 标记仍然存在，需要逐步清理（见 ACTION_PLAN.md ST-1/ST-2）
- `bump_version.py` 自动化脚本尚未建立（手动维护 VERSION 和 CHANGELOG 仍有遗漏风险）

---

## 教训

**教训 1**：版本治理应该在 v7.0 时就建立，而不是等到 v8.0。越晚引入版本治理，迁移成本越高——v7 时期积累的 500+ 版本标记，现在需要花费 4h+ 才能清理完毕。

**教训 2**：SSOT 不只是版本号的问题，而是"任何关键信息只有一个权威来源"的原则。这个原则应该更广泛地应用：配置、状态定义、路由规则——每类信息应该有一个且仅有一个权威来源。

**教训 3**：版本治理的配套工具（`bump_version.py`、CI 门禁）和规范文档（本 ADR）同等重要。没有工具支持的规范往往沦为一纸空文。
